01

What is HTML? (Concept & Origin)

HTML stands for HyperText Markup Language. It is **not a programming language** — it is a markup language. This means it describes the structure and semantic meaning of content rather than logic, flow control, or computation (no loops, no variables, no conditionals).

  • "Markup": You annotate plain content with tags to tell the browser what that content represents (a header, a paragraph, an image, or a link).
  • "Hypertext": Documents contain hyperlinks enabling users to navigate between documents instantly. This was the revolutionary foundation of the web in 1991.
History Spotlight: The CERN Origin Story

In 1989-1991, **Tim Berners-Lee** at CERN created HTML because physicists needed to share research documents across different computers running proprietary operating systems.

Before HTML, documents were formatted using incompatible word processor systems (like WordPerfect or early Word versions). They were unreadable on other systems and lacked any cross-document linking. HTML solved this by creating a simple, text-based document format that *any machine* could render, with references (hyperlinks) pointing to other files.

02

Deep Understanding: How Browsers Render HTML

To render HTML, the browser executes a multi-stage parsing pipeline that translates text files into active pixels on screen.

The Critical Rendering Path Stages
  • DOM (Document Object Model) Construction: The parser consumes HTML character streams, maps them to token tags, and structures them into a node tree representing the HTML structure.
  • CSSOM (CSS Object Model) Construction: Style sheets are tokenized and parsed into a rule map describing cascades, matches, and overrides.
  • Render Tree Merging: The DOM and CSSOM merge. Nodes that are not renderable (e.g. <head> elements, scripts) or nodes explicitly hidden via CSS (display: none) are excluded from this tree.
  • Layout (Reflow): The browser calculates the exact geometry—width, height, coordinates, and wrapping rules—for every visible render tree box on the viewport.
  • Paint: The browser rasterizes the layout shapes, filling in text characters, colors, borders, gradients, and shadows into raster image tiles.
  • Composite: Layout layers (like elements with 3D transforms or independent scrolls) are sent to the GPU to be layered and composited onto the monitor.
03

The Tokenizer — Stage 1 of Parsing

The Tokenizer reads raw HTML bytes character-by-character and slices them into small units called tokens (StartTag, EndTag, Character, Comment). Crucially, the tokenizer has no concept of nesting, hierarchy, or tree structures—it is a dumb, linear processor. Conceptually, it is a reader sounding out individual words without yet understanding the grammar of the sentence.

The Finite State Machine (FSM) trace

HTML tokenization is specified as a deterministic Finite State Machine (FSM). It shifts between states such as Data state, Tag open state, Tag name state, Attribute name state, and Attribute value state based on the characters it encounters.

For example, parsing the markup <div class="card"> triggers the following state transitions:

  • Data state (reading plain text): encounters < → switches to Tag open state.
  • Tag open state: encounters d (a letter) → detects a start tag → switches to Tag name state.
  • Tag name state: consumes div until it hits whitespace → switches to Before attribute name state.
  • Before attribute name state: encounters c → switches to Attribute name state.
  • Attribute name state: consumes class until it hits = → switches to Attribute value state.
  • Attribute value state: consumes characters inside quotes (card) → switches back to Tag open state.
  • Tag open state: encounters > → emits a StartTag: "div" token with attribute class="card" → switches back to Data state.
Tokenizer Trap: Special Characters inside Attribute Values

Consider this tag: <div class="hello>world">

A common developer misconception is that this tag closes at the first >. However, because the tokenizer is currently in the Attribute value state (having consumed an open quote "), it treats the > character as **literal text inside the attribute value**, NOT as the tag closer. The state machine will only exit the attribute-value state when it encounters the matching closing quote ".

Key Comparison: Tokenizer vs. Tree Construction
  • Tokenizer (Stage 1): Identifies individual pieces (tokens). It is linear, dumb, and operates without any concept of tag hierarchy or nesting.
  • Tree Construction (Stage 2): Consumes the emitted tokens to build the DOM tree. It enforces document hierarchy, tracks parent/child nesting, and executes recovery algorithms to auto-close unclosed tags.
04

DOM vs HTML: The Crucial Interview Distinction

Interviewers love checking if you understand the conceptual boundary between HTML files and dynamic browser memory.

  • HTML: The raw text markup stored in your static file. When you right-click on a page and select "View Page Source", you see the exact, unchanged HTML originally served by the web host.
  • DOM (Document Object Model): The live, in-memory tree object representation constructed by the browser. JavaScript mutates this tree directly (via appendChild, innerHTML, etc.) without touching the original HTML file.
Real-World Payoffs & Debugging

Next.js SSR & React Hydration: The server sends fully-formed HTML first (for fast first paint and SEO indexability). Then, React "hydrates"—attaching event listeners to that already-existing DOM instead of rebuilding it from scratch. Understanding DOM ≠ HTML explains why hydration works this way.

QA Payoff: "The page displays content X but View Source does not show X." This is because client-side JavaScript mutated the DOM after load; it is not a server rendering bug.

05

The HTML Parser: Deterministic Error Recovery

Unlike XML or JSON (which throw hard syntax errors on malformed markup), the HTML parser operates with strict, deterministic error-recovery rules.

This was a deliberate design decision: by the time HTML5 was standardized, millions of web pages had unclosed tags and nested syntax errors. The HTML5 spec formalized "how all browsers must fix this identically" so pages render consistently across different rendering engines.

Hands-On Parser Prediction Challenge

Consider this malformed markup block: 

<html>
<head><title>Test</title>
<body>
<p>First paragraph
<p>Second paragraph
</body>

What does the parser do to rebuild the tree?
1. Missing DOCTYPE: Triggers quirks mode.
2. Unclosed <head>: When the parser encounters the <body> tag, it knows a body cannot live inside a head. It automatically inserts a closing </head> tag.
3. Unclosed <p>: When it reads the second <p> tag, it knows a paragraph cannot contain nested block-level paragraphs. It auto-closes the first paragraph with a </p> before opening the second.

06

HTML Document Structure Roles

Every standard web page contains structural elements with specific browser configurations.

DOCTYPE: Standards Mode vs. Quirks Mode

Writing <!DOCTYPE html> at the top of the file triggers Standards Mode, ensuring layout calculations comply with modern CSS specs.

If you omit it, the browser falls into Quirks Mode, emulating bugs from the 1990s (like Netscape 4 box-model sizing where paddings subtract from content bounds, and incorrect font-style inheritance inside tables). This exists purely to keep decades-old websites readable.

Tag Roles & Render Blockages
  • <html lang="en">: Root node of the DOM tree. The lang attribute is crucial—it controls screen-reader voice profiles and translation triggers.
  • <head>: Metadata container. Items inside do not render directly onto the page viewport.
  • <body>: Contains all visible page markup that combines into the Render Tree. Multiple body tags are illegal; additional ones are automatically merged into the first body node.
  • Case-Sensitivity: Tag and attribute names are NOT case-sensitive (<DIV> is parsed as <div>), but lowercase is the universal convention.
  • CSS Render Blocking: The browser does NOT build the Render Tree until CSSOM is ready to prevent a Flash of Unstyled Content (FOUC). This is why render-blocking CSS in the head delays first paint.
07

Interview Questions & Answers

Q1: Is HTML a programming language? Why or why not?

Answer: No, HTML is a markup language. It describes structure and meaning of static page elements using tags. It lacks logic execution capabilities—such as runtime variables, evaluation loops, conditional expressions, or numeric calculations—which are the defining traits of programming languages.

Q2: Can a page legally have more than one <body> tag? What does the browser do if you try?

Answer: According to the W3C specification, a page can only contain a single <body>. If a document includes multiple body tags, the HTML parser's error recovery automatically merges any content and attributes inside the additional body tags into the first body node in the DOM tree, avoiding layout compilation errors.

Q3: Can you put a <script> tag inside <head>? What changes if you do vs. placing it at the end of <body>?

Answer: Yes, script tags are valid in both elements. However, if a normal script is placed in the <head>, the HTML parser stops building the DOM tree until the script is fully downloaded and executed (parser-blocking behavior), leading to a blank viewport. Placing scripts at the bottom of the <body> ensures the parser completes DOM construction first, displaying the page faster. Alternatively, scripts in the head can use defer or async attributes to prevent parser-blocking.

Q4: Does the browser wait for CSS to finish loading before building the Render Tree? Why?

Answer: Yes. While the browser builds the DOM tree incrementally as HTML stream bytes arrive, it **cannot** build the Render Tree without the CSSOM. If the browser merged styles dynamically, it would trigger a Flash of Unstyled Content (FOUC), rendering raw default styles followed by sudden layout shifts as CSS rules load. Therefore, CSS acts as a render-blocking resource.

Q5: If the tokenizer is in `Attribute value state` and hits a `>` inside an open quote, does it close the tag?

Answer: No. Inside an open quote, the state machine is in `Attribute value state`, so the > is treated as literal text inside the attribute. The tag only closes when the matching quote is found, returning the state machine back to tag open/name states.

💡 Senior Interview Follow-up Challenge
"You said the DOM is built from parsing HTML. If JavaScript uses document.write() while the HTML is still being parsed, what does that do to the parsing process — and why do modern engineers consider document.write() a performance anti-pattern?"
The Answer:
Calling document.write() while HTML is being parsed forces the parser to immediately halt, dynamically inject the written text into the token stream, re-run the tokenizer, and compile those tags on the fly before resuming parsing of the remaining document.

Why it is a performance anti-pattern:
  • Parser Blocking: It forces the browser's speculative pre-parsers (which look ahead to preload scripts and styles) to pause because the injected script content could rewrite subsequent markup. This breaks parallel resource downloading.
  • Destructive Behavior: If document.write() is executed *after* the DOM parsing is complete (for example, inside an async callback or after onload), it immediately overrides and clears the entire current document, wiping out the page.
  • Legacy Layout Shifts: Modern engines cannot optimize rendering queues when pages dynamically inject text streams into the parser mid-run.
