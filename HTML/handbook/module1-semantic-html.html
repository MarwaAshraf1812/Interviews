Module 1: Semantic HTML & Accessibility

Learn to write structure with intent: understand document layouts, the accessibility tree (AOM) translation layer, keyboard focus properties, and anchor-vs-button semantics.

01

Semantic Outlining (<article> vs <section> vs <div>)

Semantic HTML is about writing markup that describes the **meaning and purpose** of the content, not just its visual appearance. In senior interviews, a classic trap tests your choice of containers for a product card listing.

The correct semantic container for an individual product card is an <article>. Here is why:

The Semantic Grid Hierarchy
  • <article>: Represents a self-contained, independent composition. If you extract this card and place it in a search result page, a wishlist, or an RSS feed, it remains fully logical and understandable on its own.
  • <section>: Groups related contents under a single heading. A section should not be used as a styling container; it represents a thematic region of the page and typically starts with a header tag (e.g. <h2>).
  • <div>: A non-semantic container. It holds zero structural meaning and should be used strictly for layout styling (like Flexbox and Grid wrappers) or JavaScript bindings.
<!-- Perfect Semantic Structure for a Catalog Section -->
<section class="featured-products">
  <h2>Featured Products</h2>

  <div class="products-grid"> <!-- div handles only CSS Grid Layout -->
    
    <article class="product-card">
      <img src="iphone.jpg" alt="iPhone 15 Pro Max">
      <h3>iPhone 15 Pro Max</h3>
      <p>$1,199.99</p>
      <button type="button">Add to Cart</button>
    </article>

  </div>
</section>
Under the Hood: Who Determines Heading Sizes?

Does HTML natively specify that <h1> is 32px? **No.**

HTML only describes structural semantics. Every browser includes a built-in default CSS file called the User Agent Stylesheet. The browser uses this internally to apply standard styles (like h1 { font-size: 2em; font-weight: bold; }). If you delete all custom CSS, this stylesheet remains active. If you disable it completely, elements render as plain, unstyled, inline blocks of text.

02

Interactive Intent: Anchor (<a>) vs Button (<button>)

Selecting between <a> and <button> is based on **intent and behavior**, never on visual presentation.

The Rule of Thumb:
• Use <a> (Anchor) for **navigation**: taking the user to a new URL, hash, or trigger a resource download.
• Use <button> for **actions**: executing operations that alter state, open modals, or submit forms.

Trap: Destructive Actions inside Anchors

An interviewer might show you this code and ask for feedback: 

<a href="/delete-account">Delete Account</a>

Why is this dangerous?
1. **Crawlers & Prefetching:** Browsers and search engine indexing bots prefetch anchor links to improve speed. If a bot crawls the page, it will follow the link and accidentally trigger the account deletion!
2. **HTTP Semantics:** GET requests must be safe and idempotent. Actions that change database state (like deletions or additions to carts) should be triggered via POST/DELETE requests using a <button> nested inside a form, or handled via script-bound event handlers.

03

The Accessibility Tree (AOM)

Screen readers and other assistive tools do not parse the DOM tree directly. Instead, the browser translates the DOM tree into an **Accessibility Tree** using native OS accessibility APIs.

When you use semantic markup, the browser automatically maps elements to their corresponding roles, states, and names in the tree. When you use generic <div> elements, they are completely omitted or announced as un-interactive groups.

Interview formulation: Why not use divs for everything?
"Using only divs strips away all semantics. Screen readers rely on the browser's accessibility tree. When we use <button>, the browser announces its role ('button'), name, and state ('enabled/disabled') automatically. With a div, the user only hears the static text content, losing all context that the element is an interactive control."
04

Keyboard focus & tabindex mechanics

Another major difference between semantic interactive elements and custom containers (like <div> or <span>) is **Keyboard Accessibility**.

Standard buttons and links are focusable by default. Users navigating with a keyboard can use the Tab key to jump directly to them, and trigger their action using Enter or Space.

The Cost of Recreating Behaviors with Divs

If you implement a click handler on a div: 

<div onclick="buy()">Buy Now</div>

It is completely broken for keyboard users. To make it work as a button, you would have to write: 

<div onclick="buy()" tabindex="0" role="button" onkeydown="handleKey(event)">Buy Now</div>

Where tabindex="0" adds the element to the sequential tab order, role="button" sets the accessibility role, and handleKey intercepts Enter and Space keys to invoke the action. Native buttons give you all of this automatically without writing a single line of JavaScript.

tabindex Values Gotcha
  • tabindex="0": Places the element in the normal keyboard tab order based on DOM layout sequence. Use this to make custom elements focusable.
  • tabindex="-1": Removes the element from the keyboard tab sequence but allows it to be focused programmatically via JavaScript (e.g. element.focus()).
  • tabindex="> 0": **Avoid this.** It forces a custom tab order, breaking the natural sequential flow and disrupting the experience for keyboard-only users.
05

Interview Questions & Answers

Q1: Why doesn't HTML need built-in variables or loops?

Answer: HTML's primary responsibility is to describe structural metadata and content representation. It is a markup language, not a programming language. Generating data sequences or performing logical evaluations is the responsibility of programming languages (JavaScript, Server scripts, or UI frameworks). Keeping HTML markup logic-free ensures clean separation of concerns.

Q2: What happens if CSS is completely disabled in a browser?

Answer: The page elements will still render and function (text block layout, input typing, button submissions, links redirecting) based on the browser's internal default User Agent Stylesheet, but all visual presentation (position grids, responsive adjustments, color tokens, animations) will disappear.

Q3: Why is a <button> still the correct element to trigger a "POST /cart/items" request instead of an anchor?

Answer: Clicking "Add to Cart" is an action that modifies server state. In web architecture, anchor tags correspond to GET requests, which must remain safe and idempotent (devoid of side effects). Allowing bots or browsers to prefetch links that modify state causes unintended behaviors. Using a button reflects an action, manages focus, and prevents search engine crawlers from auto-triggering actions.

Q4: How does a button behave differently inside a form element compared to a div?

Answer: Inside a <form>, a standard <button> defaults to type="submit". Pressing the Enter key inside any text input in that form triggers the button's click handler and submits the form data automatically. A <div> holds no association with form submission sequences and will not respond to inputs unless manually wired.

💡 Mini Quiz & Concept Check
If you have a clickable product card title that redirects users to the product details page, would you wrap the title in a <button> or an <a> tag? Why?
Answer: An anchor <a href="/product/details">.

Explanation: Clicking the title navigates the user to a different URL (a new resource path). This represents navigation, which is the primary semantic purpose of an anchor tag. Even though we can style a button to look like a text link using CSS, using a button would break standard anchor navigation features, such as the ability to right-click to open in a new tab, search index bookmarking, and keyboard history redirection.

HTML Semantic Sections — Interview Traps & Tricky Questions

This section analyzes common interview pitfalls, senior-level traps, and strategic questions regarding semantic elements.

The Core Decision Framework (Say this out loud in interviews)
Could it stand alone, independent of the page?  →  <article>
Is it a themed chunk that's part of a larger whole?   →  <section>
Pure layout/styling hook, no semantic meaning?      →  <div>

Common Traps & Pitfalls

Trap 1 — <section> without a heading
A <section> represents a thematic group and is expected to contain a heading (h1–h6). A section with no heading is a layout element; use a <div> instead. 
<!-- Wrong: pure layout wrapper -->
<section class="flex-wrapper">
  <button>Save</button>
</section>

<!-- Right -->
<div class="flex-wrapper">
  <button>Save</button>
</div>
Trap 2 — Using <section> as a generic styling container
Do not use <section> for flexbox/grid containers or wrappers just because it "sounds more semantic than div." Reach for <section> only when there's thematic relevance.
Trap 3 — <article> vs <section> for repeatable cards
If listing items that could stand alone (e.g. comments, forum posts, product cards), wrap each item in an <article>, and the group itself in a <section>. 
<section aria-label="Comments">
  <h2>Comments (42)</h2>
  <article class="comment">
    <h3>Ahmed</h3>
    <p>Great article!</p>
  </article>
</section>
Trap 4 — More than one <main>
There must be **exactly one** visible <main> tag per page. It must never be nested inside <article>, <aside>, <header>, <footer>, or <nav>.
Trap 5 — Skipping heading levels
Jumping from <h1> to <h3> breaks document layout structures for screen readers. Heading hierarchy determines accessibility navigation, not styling. Use CSS for font sizing.
Trap 6 — <nav> for every list of links
Reserve <nav> only for major navigation blocks (primary, pagination, breadcrumbs). A minor group of related links inside a post footer should just use standard unordered lists.
Trap 7 — <header> and <footer> are NOT page-only
Each <article> or <section> can contain its own scoping <header> and <footer> tags. 
<article>
  <header>
    <h2>Blog Title</h2>
  </header>
  <p>Content...</p>
  <footer>Tags: HTML</footer>
</article>
Trap 8 — <aside> doesn't just mean "sidebar"
<aside> represents content tangentially related to the surrounding page. It can render visually anywhere (e.g. pull-quotes, inline glossary tags), not just side columns.
Trap 9 — Over-semantic layouts
Avoid wrapping layout tags in articles/sections unnecessarily. <div> remains the correct choice for structural containers with no semantic value.
Trap 10 — The HTML5 outline algorithm is dead
Browsers and assistive tools never implemented heading outlines reset inside sections. Always maintain explicit manual heading progression (h1 → h2 → h3).

Interview Questions by Level

Beginner:
  • What's the difference between <div> and <section>?
  • Why do we use semantic tags instead of just <div> everywhere?
  • Can a page have more than one <h1>?
  • What does the <main> tag do, and how many can a page have?
Intermediate:
  • Mapping structures: News page with hero card, previews grid, and trending topics. Assign semantic tags.
  • Why should <label for="..."> link to inputs if CSS styling is custom?
  • Is it valid to place <header> inside <article>? Explain.
  • What is wrong with wrapping every layout element in <section>?
Senior:
  • How does a screen reader user navigate landmark elements vs. div soup? What shortcuts are lost?
  • Explain why the HTML5 outline algorithm was dropped and what heading layout pattern must be used.
  • A junior on your team wrapped modals and toasts in <article>. Review this PR layout.
  • When is <div role="button"> correct, and what focus / keyboard accessibility requirements are required?

Rapid-Fire Cheat Sheet

Situation Correct Tag
Product card, blog post, comment, tweet <article>
Themed group with a heading <section>
Pure layout wrapper, no meaning <div>
Primary / footer / breadcrumb navigation <nav>
Related-but-separate content (sidebar, callout) <aside>
Page's single primary content block <main>
Intro / Outro block for page or article <header> / <footer>
