01

Equality Comparison

In JavaScript comparisons, we have two primary types: Loose Equality (==) and Strict Equality (===).

Strict Equality (===) directly compares both the value and the type of the operands, without performing any data type modification. In contrast, Loose Equality (==) performs implicit Type Coercion (automatically converting data types) to force the values into the same type before comparison. This conversion follows a strictly defined standard known as the Abstract Equality Comparison Algorithm. 

// Strict Equality (===) - No Coercion
const a = 5;
const b = "5";
console.log(a === b); // false (number vs string)

// Loose Equality (==) - Coercion happens
console.log(a == b);  // true (string gets converted to number)
Under the Hood: Abstract Equality Algorithm
According to ECMAScript spec section 7.2.14:
When comparing x == y:
  1. If Type(x) is the same as Type(y), return x === y.
  2. If x is null and y is undefined, return true.
  3. If x is undefined and y is null, return true.
  4. If Type(x) is Number and Type(y) is String, compare x == ToNumber(y).
  5. If Type(x) is String and Type(y) is Number, compare ToNumber(x) == y.
  6. If Type(x) is Boolean, compare ToNumber(x) == y.
  7. If Type(x) is Object and Type(y) is either String, Number, or Symbol, compare ToPrimitive(x) == y.

The Special Case: null == undefined

This is one of the most debated rules in JavaScript. Why does null == undefined yield true? The reason is simply that the ECMAScript Specification defines this as an explicit, special rule in the algorithm (steps 2 and 3 above). The engine does not convert them to numbers here (meaning null is not converted to 0, and undefined is not converted to NaN). The spec states: because both represent the "absence of value", they should be considered equal under loose equality.

However, null === undefined evaluates to false because their types are strictly different (the first is of type Null, and the second is of type Undefined).

Interview Trap

Trap: An interviewer might ask: "Does null == 0 evaluate to true?".
Many developers incorrectly guess "Yes, because null coerces to 0". Correct Answer: ❌ False! The ECMAScript specification explicitly states that null and undefined do not equal any other values besides each other under loose equality. No coercion is performed on null to make it a number when compared to 0.

Interview-Ready Answer
"In JavaScript, loose equality (==) allows implicit coercion according to the Abstract Equality Comparison Algorithm, whereas strict equality (===) compares both value and type without coercion. For example, null == undefined evaluates to true because the ECMAScript spec explicitly defines them as equal. However, null === undefined is false because null has type 'Null' and undefined has type 'Undefined'."
💡 Mini Quiz
What is the output of: [] == ![]?
Answer: true

Step-by-step Execution:
  1. ![] executes first (logical NOT). Since an empty array is truthy, ![] becomes false. Now we have: [] == false.
  2. If one side is Boolean, convert it to a Number. false becomes 0. Now we have: [] == 0.
  3. If one side is Object (Array) and the other is Primitive, call ToPrimitive([]). For arrays, this calls toString() which results in "". Now we have: "" == 0.
  4. If comparing String to Number, convert String to Number. Number("") is 0. Now we have: 0 == 0.
  5. Both sides are now numbers, so it returns true.
02

NaN (Not a Number)

The value NaN (Not a Number) is produced when attempting to perform an invalid mathematical operation (such as dividing 0 by 0, or subtracting a string from a number).

The most critical rule regarding NaN is: NaN is the only value in JavaScript that does not equal itself! That means both NaN == NaN and NaN === NaN evaluate to false. 

// Producing NaN
const x = 0 / 0;      // NaN
const y = "hello" - 5; // NaN

console.log(x === y); // false
console.log(NaN === NaN); // false
Under the Hood: IEEE-754 Standard & Floating Point Arithmetic
JavaScript represents numbers as double-precision floating-point format (64-bit) defined by the IEEE 754 standard. Under IEEE 754, NaN is represented with an exponent field of all 1s and a non-zero fraction. The standard explicitly states that:
"Any comparison operation involving NaN, whether strict or non-strict, must evaluate to false."
This is because NaN represents an undefined/invalid mathematical operation, and two failed calculations aren't logically identical (e.g. 0/0 is not equivalent to Math.sqrt(-1)).

isNaN() vs Number.isNaN()

Global isNaN(): First attempts to perform Type Coercion on the input, converting it to a number. If the conversion fails and returns NaN, it returns true. Consequently, isNaN("abc") evaluates to true because "abc" coerces to NaN.
Strict Number.isNaN(): Does not perform any type coercion. It asks a single, strict question: "Is the type of the value Number, and is the value currently NaN?". Therefore, Number.isNaN("abc") evaluates to false because "abc" is a string.

Interview Trap & Common Bugs

Trap: Using the global isNaN(x) to validate user input.
If the input is undefined, isNaN(undefined) returns true (because undefined coerces to NaN). However, using Number.isNaN(undefined) correctly returns false.
Classic Interview Trick: How can you check if a value is NaN without using any helper functions?
Answer: Use value !== value. If this check evaluates to true, it guarantees that the value is NaN.

Interview-Ready Answer
"NaN represents an invalid mathematical operation under the IEEE 754 standard. It is unique because it is the only value in JavaScript that does not equal itself. We should never use the global isNaN() function to check for NaN because it performs type coercion and incorrectly flags values like 'abc' or undefined as NaN. Instead, we should use the strict Number.isNaN(), or the self-comparison trick: value !== value."
💡 Mini Quiz
What is the output of the following checks? 
console.log(isNaN(undefined));
console.log(Number.isNaN(undefined));
Output:
true
false

Explanation:
The global isNaN() coerces undefined to a number. Number(undefined) is NaN, so it returns true.
Number.isNaN() checks strictly. Since undefined is of type Undefined (not Number), it immediately returns false without coercing.
03

typeof Operator

The typeof operator is used to evaluate the data type of a value. It always returns a string representing the type.

Under the Hood: Every Possible Return Value & The null Bug
The only possible strings typeof can return are:
  • "undefined" (for undefined values)
  • "boolean" (for booleans)
  • "number" (for numbers)
  • "string" (for strings)
  • "symbol" (for ES6 Symbols)
  • "bigint" (for ES2020 BigInts)
  • "object" (for arrays, null, dates, regex, plain objects)
  • "function" (for callable objects/functions)

The Historical typeof null Bug:

In the initial 1995 release of JavaScript, values were stored in memory as 32-bit units, composed of a type tag (the first 1-3 bits) and the actual value. The type tag for Objects was 000. The null pointer was represented entirely as zeros (0x00000000). The engine read the leading bits, saw 000, and immediately assumed that null was an object, returning "object"! This historical bug has been preserved in the specification because fixing it would break millions of legacy websites built on this behavior.

JS memory cell layout (32-bit tag system in 1995): ┌───────────┬────────────────────────────────────────┐ │ Tag (3b) │ Value / Pointer bits │ └───────────┴────────────────────────────────────────┘ 000 -> Object pointer 001 -> Int number 010 -> Double number Null representation: ┌───────────┬────────────────────────────────────────┐ │ 000 (tag) │ 00000000000000000000000000000 (value) │ └───────────┴────────────────────────────────────────┘ Tag matches "000" (Object) -> typeof null evaluates to "object" ❌ (Spec bug)

Why does typeof function return "function"?

Even though functions are objects under the hood, the engine returns "function" to easily identify them as callable objects that implement the internal [[Call]] method.

Interview-Ready Answer
"The typeof operator returns a string indicating the type of the unevaluated operand. There are exactly 8 possible return values. typeof null returns 'object' due to a 1995 design oversight where JavaScript's binary representation used type tags, and 000 was reserved for objects. Because null was represented as all zeros, the engine mistakenly identified it as an object. This remains in the spec to maintain backward compatibility."
💡 Mini Quiz
What is the output of the following expressions? 
console.log(typeof typeof 42);
console.log(typeof [1, 2, 3]);
Output:
"string"
"object"

Explanation:
1. typeof 42 evaluates to the string "number". Then, typeof "number" evaluates to "string".
2. Arrays are objects under the hood, so typeof [] evaluates to "object".
04

instanceof vs Array.isArray

The instanceof operator tests whether the prototype property of a constructor function appears anywhere in the prototype chain of an object. 

class Animal {}
class Dog extends Animal {}

const myDog = new Dog();

console.log(myDog instanceof Dog);    // true
console.log(myDog instanceof Animal); // true (walks up prototype chain)
console.log(myDog instanceof Object); // true
Under the Hood: Prototype Chain Walk Algorithm
When checking x instanceof Constructor:
  1. First, check if x is an Object. If it's a Primitive (like a string literal "hello"), return false.
  2. Get the prototype of x: let proto = Object.getPrototypeOf(x) (internally x.__proto__).
  3. Loop:
    • If proto === null, return false.
    • If proto === Constructor.prototype, return true.
    • Set proto = Object.getPrototypeOf(proto) and repeat.

Why Array.isArray() is superior to instanceof Array

In web browsers, when working with iframes, each iframe has its own separate execution environment (Execution Realm), meaning it has its own Array constructor. If you pass an array from an iframe to the main page and check arr instanceof Array, it will return false because the array's prototype belongs to the iframe's Array constructor, not the parent window's Array constructor.
This is why Array.isArray() is superior: it checks the internal structure of the object and accurately identifies it as an array regardless of which realm it originated from.

Interview Trap & Common Mistakes

Trap: Writing "hello" instanceof String.
The result is ❌ false! This occurs because "hello" is a primitive string, not an object created via the new String() constructor. The instanceof operator always returns false for primitives.

💡 Mini Quiz
What does the following print? 
const s = new String("hello");
console.log(s instanceof String);
console.log("hello" instanceof String);
Output:
true
false

Explanation:
The first evaluates to true because s is an object explicitly instantiated from the String constructor.
The second is false because primitive strings are not instances of constructors in the prototype chain check.
05

Operators Overview

Operators form the core syntax of JavaScript. Here is a brief structured overview:

  • Arithmetic: Operations like addition, subtraction, division, modulus, and exponentiation (+ - * / % **).
  • Comparison: Standard checks and equality comparisons (> < >= <= == === != !==).
  • Logical: Boolean combination operators (&& || !).
  • Unary: Operators acting on a single operand (+ - typeof delete void !).
  • Assignment: Storing values with arithmetic shorthands (= += -= *= /= %=).
  • Bitwise: Performing binary bit-level operations (& | ^ ~ << >> >>>).
// Unary + is a quick way to convert a string to number
console.log(+"42"); // 42 (Number)

// Arithmetic Exponentiation
console.log(2 ** 3); // 8 (2 cubed)
06

Operator Precedence

Operator Precedence dictates the order in which operators are evaluated when they appear in a single expression without explicit parentheses.

Precedence Table (Most Important for Interviews)
Listed from highest priority (19) to lowest priority (2):
Precedence Operator Category Associativity
19 ( ) (Parentheses) n/a
18 . [ ] new (Member Access) Left-to-Right
15 ! typeof unary + - (Unary) Right-to-Left
14 ** (Exponentiation) Right-to-Left
13 * / % (Multiplication) Left-to-Right
12 + - (Addition) Left-to-Right
9 == === (Equality) Left-to-Right
5 && (Logical AND) Left-to-Right
4 || ?? (Logical OR / Nullish) Left-to-Right
2 = += -= (Assignment) Right-to-Left

Associativity

When two operators share the same precedence level, their evaluation order is determined by their associativity.
Right-to-Left Associativity: Seen in assignment operators like a = b = c (evaluates b = c first, then assigns to a) and exponentiation 2 ** 3 ** 2 (which acts as 2 ** (3 ** 2) yielding 512, rather than (2 ** 3) ** 2 yielding 64).

💡 Mini Quiz
Evaluate this expression step by step: 
let x = 1;
let y = typeof x + "2";
console.log(y);
Answer: "number2"

Explanation:
According to precedence, typeof (priority 15) is higher than addition/concatenation + (priority 12).
So typeof x evaluates first to the string "number".
Then "number" + "2" evaluates to "number2".
07

Logical Operators & Short-circuiting

Logical operators like &&, ||, and ! leverage an optimization pattern called Short-circuit evaluation.

The engine stops evaluating the expression as soon as the final result can be determined:
• && stops at the first falsy value it encounters and returns it. If none are falsy, it returns the last value.
• || stops at the first truthy value it encounters and returns it. If all are falsy, it returns the last value. 

// && Short-circuiting
console.log(false && sideEffect()); // false (sideEffect never runs)
console.log("hello" && 42);         // 42 (returns last value)

// || Short-circuiting
console.log("hello" || false);      // "hello" (stops at first truthy)
Under the Hood: Side Effects & Guard Clauses
Because JavaScript short-circuits, developers use it to implement Guard Clauses to prevent runtime errors:
user && user.logout();
If user is null/undefined, the expression evaluates to null/undefined and stops. If it didn't short-circuit, the engine would evaluate user.logout() and crash with a TypeError.
💡 Mini Quiz
What is the output of the following checks? 
console.log(0 && "hello");
console.log("hello" || "world");
console.log(null || 0 || "react" && "next");
Output:
0
"hello"
"next"

Explanation:
1. 0 is falsy, so && stops immediately and returns 0.
2. "hello" is truthy, so || stops immediately and returns "hello".
3. "react" && "next" evaluates first due to higher precedence of && over ||. Since "react" is truthy, it returns "next". The expression becomes: null || 0 || "next".
null is falsy (continue), 0 is falsy (continue), "next" is truthy, so it returns "next".
08

Nullish Coalescing (??)

The Nullish Coalescing (??) operator was introduced in ES2020 to resolve a major design limitation of ||.
While || falls back on any falsy value, ?? only falls back when the left-hand side is strictly null or undefined. 

// Difference between || and ??
const userRetries = 0;

console.log(userRetries || 3); // 3 (0 is falsy, falls back)
console.log(userRetries ?? 3); // 0 (0 is not nullish, keeps 0)
Interview Trap: Mixing ?? with && / || without Parentheses

Trap: Writing const x = a || b ?? c; will immediately throw a ❌ SyntaxError!
The language specifications explicitly prohibit chaining ?? with && or || without parentheses. This prevents ambiguity regarding which operator should take precedence, keeping the code predictable. 

const x = (a || b) ?? c; // Correct ✅
💡 Mini Quiz
What is the output of: console.log(false ?? true)?
Answer: false

Explanation:
Since false is not null or undefined, the nullish coalescing operator immediately evaluates to the left-hand side, keeping false.
09

Real-World Bugs: || vs ??

In production applications, relying on || to configure default values is a very common source of silent, logical bugs.

Real Bug Scenario 1: Volume Slider
Suppose you build a game or video player setting volume. A volume of 0 is a valid user action (mute). 
const userVolume = 0;
const finalVolume = userVolume || 0.5;

❌ Bug: The player will automatically apply 0.5 because 0 is falsy! The user will never be able to mute the player.
✅ Solution: Use userVolume ?? 0.5;

Real Bug Scenario 2: Dark Mode Switch
A user explicitly turns OFF dark mode (sets preference to false). 
const preferences = { darkMode: false };
const useDarkMode = preferences.darkMode || true;

❌ Bug: The website will remain in dark mode forever because false is falsy!
✅ Solution: Use preferences.darkMode ?? true;

10

Interview Questions & Answers

Here is a compilation of critical, deep-dive interview questions regarding these fundamental concepts:

Q1: Why does typeof null return "object" and how can we check if a value is strictly null?

Answer: It is a historical bug from 1995 caused by the binary type tag allocation for objects (000) matching null's binary pointer (all zeros). To check if a value is strictly null, we should use strict equality: value === null.

Q2: Why does NaN === NaN evaluate to false, and what standard governs this behavior?

Answer: This behavior is governed by the IEEE 754 standard for floating-point numbers. Any comparison with NaN is defined to be false because NaN represents an undefined or unrepresentable numerical result, and two distinct invalid calculations are not mathematically identical.

Q3: Explain step-by-step why [] == ![] returns true.

Answer: First, ![] evaluates to false. Then, [] == false coerces the boolean to a number, yielding [] == 0. Next, ToPrimitive([]) runs, calling [].toString() which results in "" == 0. Finally, the empty string is coerced to a number: Number("") == 0 which is 0 == 0, yielding true.

Q4: What is the difference between typeof and instanceof?

Answer: typeof is a unary operator that returns a string representing the primitive type of the operand. instanceof checks the prototype chain of an object to verify if the prototype property of a constructor function exists anywhere in it. It returns a boolean and does not work on primitive values.
