Chapter 2: Scopes, Contexts & Closures

Master the runtime engine: understand execution context mechanics, scope chain lookups, variable environment generation, this bindings, and closures.

01

Nullish vs. Falsy Fallbacks

Before moving deeper into execution environments, let's establish clean clarity on one of the most critical coding traps in standard applications: using || vs ?? for configuring default variables.

Interview-Ready Formulation
"The nullish coalescing operator (??) only falls back to the default when the value is strictly null or undefined — it ignores every other falsy value. The logical OR operator (||) falls back on any falsy value — 0, "", false, NaN, null, and undefined. If a config parameter like PORT is legitimately the string "0" or an empty string, || would incorrectly overwrite it with the default. ?? respects that the falsy value was intentionally set, and only kicks in when the value is truly missing." 
// Concrete Bug Example
const config = { retries: 0, timeout: "" };

const retries = config.retries || 3;   // 3  ← BUG! 0 was intentional, overwritten
const retries2 = config.retries ?? 3;  // 0  ← Correct! 0 is preserved

const timeout = config.timeout || 5000;  // 5000 ← BUG! "" was deliberate, overwritten
const timeout2 = config.timeout ?? 5000; // ""   ← Correct!
Interview Strategy Recommendation
When asked about this in an interview, always lead with the general rule (the concept of Falsy vs. Nullish values) before showing Port/Timeout examples. Highlighting this general rule signals that you understand the underlying runtime specifications, rather than just memorizing standard templates.
02

Execution Context (Global vs. Function)

The Workspace Analogy: Think of an execution context like a workspace created before you start working on a project. Before the JavaScript engine executes a single line of code, it allocates a desk (memory for variables), lays out your tools (function declarations), and defines what "you" represents in that room (the this reference).

The engine relies on Execution Contexts to manage where code runs, track active scope environments, and maintain call dependencies. There are two primary types of execution contexts:

  • Global Execution Context (GEC): Created once when your script first runs. It generates the global object (window in browsers, global in Node.js) and sets this to match.
  • Function Execution Context (FEC): Created dynamically every single time a function is called (even if the same function is invoked repeatedly).
Under the Hood: The Lifecycle Phases
Every execution context has a two-phase lifecycle:
  1. Creation Phase (Allocation):
    • Sets up the Variable Environment: memory is allocated for variables declared with var (set to undefined), and function declarations (fully hoisted in memory). Variables declared with let/const are allocated memory but left unitialized (Temporal Dead Zone).
    • Builds the Scope Chain (references to surrounding lexically defined environments).
    • Determines the value of the this binding.
  2. Execution Phase (Evaluation): The engine runs the source code line-by-line, assigning values to variables and invoking function calls. 
// Hoisting in the Creation Phase
console.log(a);   // undefined (allocated and initialized to undefined)
console.log(foo); // [Function: foo] (fully hoisted in Variable Environment)

var a = 10;
function foo() { console.log("hi"); }

foo(); // Creates a new Function Execution Context

Interview Trap: Let vs Var Hoisting

An interviewer might ask: "Does let hoist?".
Yes! Both var and let are hoisted (meaning the engine allocates memory slot bindings for them in the creation phase). However, var is initialized immediately with undefined. In contrast, let/const are left uninitialized (having no value binding). The region between scope entry and actual initialization line is the Temporal Dead Zone (TDZ). Accessing the variable in this zone throws a ReferenceError.

03

Scope Chain & Lexical Scoping

The Nested Rooms Analogy: Think of scopes like nested rooms in a house. If you are standing in the kitchen (the innermost scope) and ask for the salt (a variable), you look in the kitchen first. If it's not there, you look in the living room (the surrounding parent scope). If it's still not there, you check outside (the global environment). You can look outward/upward, but people standing outside the house cannot look inside to find variables local to the kitchen.

JavaScript uses Lexical Scoping (also called Static Scoping). This means variable access boundaries are determined entirely by where the function was written in the source code, not where it is invoked at runtime. 

const globalVar = "I'm global";

function outer() {
  const outerVar = "I'm in outer";

  function inner() {
    const innerVar = "I'm in inner";
    console.log(innerVar);  // found locally
    console.log(outerVar);  // walks up to outer's scope
    console.log(globalVar); // walks up to global scope
  }

  inner();
}
outer();
Classic Trap: Lexical vs. Dynamic Scope Lookup

Consider this common question: 

const value = "global";
function first() {
  console.log(value);
}
function second() {
  const value = "local";
  first();
}
second(); // What does this log?

It logs "global", not "local". Since first() is defined globally, its parent scope is the global execution context. It has no structural link to the execution environment of second(), despite being called from inside it. Lexical scope chain lookups look at placement, not call path.

💡 Scope Walkthrough Quiz
What does this code log, and how does the lookup happen? 
let x = "outer";
function printX() {
  console.log(x);
}
function run() {
  let x = "inner";
  printX();
}
run();
Answer: "outer"

Step-by-step lookup process:
  1. run() is executed, generating its local variable environment containing x = "inner".
  2. Inside run, printX() is called. This triggers the creation of an FEC for printX.
  3. The scope chain of printX is fixed lexically: printX scope ──▶ Global scope. It does not look at run's scope because printX was declared at the global level.
  4. Inside printX, the lookup for x fails locally, so the engine climbs the lexical scope chain to the global scope where it finds x = "outer" and logs it.
04

this Binding (The 5 Rules)

The Pronoun Analogy: Think of the this keyword like the word "I" in a sentence. Its meaning depends entirely on who is speaking, not where the sentence was written down. this is determined at runtime based on the call site and invocation pattern of the function.

The 5 Rules of this (Priority Order: Highest to Lowest)
  1. New Binding (Constructor function): When called with the new keyword, this points to the newly allocated object instance. 
    function User(n) { this.name = n; }
const u = new User("Marwa"); // this = u
  2. Explicit Binding (call, apply, bind): We directly specify what this should refer to. 
    greet.call(targetObj, args); // this = targetObj
  3. Implicit Binding (Method on object): When called as an object method, this binds to the object to the left of the dot. 
    myObj.greet(); // this = myObj
  4. Default Binding (Global execution/plain invocation): In a standalone function call, this binds to the global object (window in browsers) or evaluates to undefined in strict mode ("use strict").
  5. Arrow Functions (Lexical Context): Arrow functions do not declare a this binding slot. They look up this lexically via the scope chain from the enclosing scope.
Interview Trap: Callback Method Detachment

Passing an object method as a callback causes the method to lose its implicit reference: 

const obj = {
  name: "Marwa",
  greet() { console.log(this.name); }
};
setTimeout(obj.greet, 100); // undefined!

Why? obj.greet is passed as a bare function reference, detached from the parent object. When called later inside setTimeout, default binding applies.
Fix 1: Use an arrow wrapper: setTimeout(() => obj.greet(), 100)
Fix 2: Bind explicitly: setTimeout(obj.greet.bind(obj), 100)

💡 Mini Quiz: Method vs Arrow this
What does this log, and why? 
const obj = {
  name: "Marwa",
  regularGreet: function() {
    console.log(this.name);
  },
  arrowGreet: () => {
    console.log(this.name);
  }
};

obj.regularGreet();
obj.arrowGreet();
Output:
"Marwa"
undefined (or the name of the global scope object)

Explanation:
1. obj.regularGreet() uses implicit binding. The function is invoked on obj, so this resolves to obj and logs obj.name ("Marwa").
2. obj.arrowGreet() is an arrow function. It has no this. It inherits this from the surrounding lexical scope where it was defined. The surrounding scope of the obj literal declaration is the global scope (object literals do not establish a local scope block). In the global scope, this is the global object, which does not have a name property defined, resulting in undefined.
05

Closures

The Backpack Analogy: Imagine a function like a traveler. When a function is created inside another function, it packs a virtual backpack containing all the variables from its surrounding scope that it might need later. Even after the parent function completes execution and leaves the stack, the child function carries that backpack wherever it is passed.

A Closure is the combination of a function bundled together with references to its surrounding state (its lexical environment). Closures allow functions to maintain private state, encapsulation, and persistent variables without cluttering global namespaces. 

function makeCounter() {
  let count = 0; // private state variable
  return function() {
    count++;
    return count;
  };
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3 (state is preserved in the closure)
Under the Hood: Memory & Garbage Collection
Normally, when a function completes execution, its execution context is popped off the Call Stack and garbage collected.
However, when a nested function is returned or passed out, it retains a internal property [[Scope]] which holds a reference to its lexical environment. Because a live reference to the outer variable (e.g. count) still exists inside the returned function, JavaScript's Garbage Collector is prevented from cleaning it up, keeping it alive in memory.

The Classic Loop-with-Var Problem

This is arguably the single most common closure-related question asked in technical interviews. 

// The famous bug:
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Logs: 3, 3, 3 (not 0, 1, 2!)

Why does this happen? Variables declared with var are function-scoped (or globally scoped if not inside a function), not block-scoped. In this loop, there is only one physical instance of the variable i allocated in memory. The three timeout callback closures all reference this same single variable. By the time the call stack clears and the callbacks run (after 100ms), the loop has fully finished, leaving i equal to 3.

How to Fix It
Fix #1: Use let (ES6 approach): 
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
} // Logs: 0, 1, 2
Explanation: let is block-scoped. Each loop iteration creates a new, separate lexical binding for i. Thus, each callback closure captures its own distinct instance of i.

Fix #2: Use an IIFE (Pre-ES6 approach): 
for (var i = 0; i < 3; i++) {
  (function(j) {
    setTimeout(() => console.log(j), 100);
  })(i);
}
Explanation: An Immediately Invoked Function Expression (IIFE) creates a new function execution context on each iteration, freezing the current value of i by passing it as argument j.
07

Common Mistakes & Pitfalls

Here are key edge cases and performance considerations to keep in mind when utilizing closures:

  • Closures reference variables, not values: A closure does not copy the value of a variable at creation time. It points directly to the live reference. If the value is mutated later, the closure will read the mutated value.
  • Memory Leaks: Because closures prevent enclosing variables from being garbage collected, storing closures referencing large objects or unused DOM elements can lead to memory leakages if not cleaned up.
Relevance to Modern Web Stack
  • React Hooks (Stale Closures): In React, useEffect, useCallback, and event listeners capture variables from the current render context. If the dependency array is configured incorrectly, they will reference outdated variable states — a classic "stale closure" bug.
  • Express Middleware: Middleware factory patterns use closures to capture initialization config:
    function rateLimiter(limit) { return (req, res, next) => { /* accesses limit via closure */ } }
08

Interview Questions & Answers

Test your preparation with these common conceptual questions:

Q1: What will the following code print?

function createFunctions() {
  const funcs = [];
  for (let i = 0; i < 3; i++) {
    funcs.push(() => console.log(i));
  }
  return funcs;
}
const fns = createFunctions();
fns.forEach(fn => fn());

Answer: It prints 0, 1, and 2. Because i is declared using let, it is block-scoped. Each iteration of the loop creates a distinct binding of i. When the closures inside funcs are executed, they correctly resolve their respective block-scoped values.

Q2: What is a stale closure in React?

Answer: A stale closure occurs when a hook (like useEffect) captures variables from a past render state and does not update when state dependencies change. This happens if you omit dynamic variables from the dependency array, causing the hook callback closure to reference old variables.
