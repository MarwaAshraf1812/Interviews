Chapter 7: Robust Error Handling

Master structural exception handling in modern JavaScript: explore ES6 class-based inheritance for custom errors, inspect call stack and V8 stack trace mechanics, and contrast sync try/catch boundaries with event-driven async Promise rejections.

01

Custom Error Classes

The Specialized Medical Tag Analogy: Think of custom error classes like specialized medical warning tags. Instead of a generic tag that just says "Problem" (a base Error class), you have specific tags like "CardiacAlert" or "AllergyAlert" (custom error classes). This lets medical staff immediately know the type of emergency and route the patient to the right specialist (catch block) without having to inspect long textual messages, while still retaining the patient's full medical history (the call stack trace).

Why they exist: In complex production setups, catch blocks need to react differently depending on the root cause of a failure. Performing fragile string-matching on err.message (e.g. if (err.message.includes("database"))) is a major anti-pattern that breaks easily when messages change. Extending the native Error constructor allows catch blocks to filter errors securely using the instanceof operator. 

class DatabaseError extends Error {
  constructor(message, query) {
    super(message); // MANDATORY: Sets up message and captures stack trace
    this.name = 'DatabaseError'; // Overrides default "Error" name
    this.query = query; // Custom context metadata
  }
}

try {
  throw new DatabaseError('Query failed', 'SELECT * FROM users');
} catch (err) {
  if (err instanceof DatabaseError) {
    console.error(`DB Error: ${err.message} on query: ${err.query}`);
  } else {
    throw err; // Bubble up other errors
  }
}
Core Mechanics: The Super Constructor & Stack Trace Retention

When creating a custom class that inherits from Error, calling super(message) is the absolute first step inside the constructor. Under the hood, the base Error constructor executes engine-specific operations (like V8's Error.captureStackTrace) to grab the current state of the execution call stack. If you omit super():

  • A ReferenceError is thrown during construction because this is uninitialized before calling the base constructor.
  • Properties like message and stack are never created or mapped properly on the instance.

Overriding this.name: Although the instanceof check relies strictly on prototype chain references (which is unaffected by instance properties), standard serialization methods (like err.toString() or JSON encoders) look for the name property. If you forget to set this.name = 'MyCustomError', your custom errors will log to production consoles with the generic label "Error: ...", causing confusion in triage alerts.

💡 Mini Quiz: instanceof vs. name overrides
If you throw an instance of DatabaseError but forget to write this.name = 'DatabaseError' in its constructor, will err instanceof DatabaseError still evaluate to true inside a catch block?
Answer: Yes. The instanceof operator checks prototype chain linkages. It walks the prototype chain of the thrown instance (err.__proto__) and compares it with the prototype property of the class (DatabaseError.prototype). Since the class declaration sets up the prototype chain automatically, this check works perfectly regardless of any instance properties. However, if you log the error (e.g. console.log(err.toString())), it will serialize as "Error: Query failed" instead of "DatabaseError: Query failed" because the name default falls back to the parent class.
02

Async Error Handling (Rejections)

The Teleporting Net Analogy: Think of a try/catch block around synchronous code like a safety net positioned directly under a tightrope walker. If they slip and fall right there, the net catches them immediately. But if the walker later teleports to a different stage entirely (asynchronous execution, running later in a separate tick of the event loop), the original net is useless — by the time they fall, the net is no longer underneath them.

The Event Loop mismatch: A standard try/catch block can only capture exceptions thrown synchronously, while the code is actively executing within the boundaries of that block on the call stack. Once an async callback (like setTimeout or a network I/O complete event) is delegated to the Web API and scheduled for later, the try/catch block finishes execution, gets popped off the stack, and is gone. Any error thrown inside the deferred callback runs on a clean call stack and crashes as an uncaught exception.

The Uncaught setTimeout Trap

Notice why this code fails to handle the exception, causing a production crash: 

try {
  setTimeout(() => {
    throw new Error("Boom!"); // Thrown later inside a separate tick of the Event Loop
  }, 100);
} catch (err) {
  console.log("Caught:", err.message); // NEVER RUNS — the try/catch block has already finished executing!
}
How Promises and async/await Solve This

Promises solve async error routing by wrapping the async callback operation in a container that supports two distinct resolution pathways: resolve and reject. Instead of relying on stack-dependent throw constructs, callbacks report failures by calling reject(error). This routes the failure directly to any callback registered via .catch().

When using async/await, the JS compiler transforms the code. The await keyword pauses the async function execution, preserving its local variables and execution state in heap memory. When the awaited promise is rejected, the engine resumes the function context and re-throws the rejection directly onto the active stack frame. This enables standard synchronous-looking try/catch syntax to intercept async errors: 

async function fetchDashboardData() {
  try {
    const rawData = await fetchFromDatabase(); // Rejection is re-thrown as a stack exception
    console.log(rawData);
  } catch (err) {
    console.log("Caught async error:", err.message); // Works perfectly!
  }
}
The Silent forEach Async Leak

Passing an async function to Array.prototype.forEach is a dangerous mistake: 

async function processArray(items) {
  items.forEach(async (item) => {
    await riskyDatabaseWrite(item); // If this throws, the rejection goes unhandled!
  });
}

Because forEach executes the callback synchronously and is completely oblivious to promises, it does not await the callback, does not collect the returned promises, and does not propagate exceptions. If a write fails, it leaks as an unhandled promise rejection. To process loops sequentially with error boundaries, always use for...of loops instead.

03

Interview Questions & Answers

Master the concepts and prepare for senior JavaScript interviews:

Q1: Why does forgetting super(message) in a custom error constructor break the instance?

Answer: Under ES6 class specifications, subclass constructors must invoke super() before accessing this. If omitted, the engine throws a ReferenceError during instantiation because the subclass prototype is not linked to the base constructor's instance creation cycle. Furthermore, the parent Error constructor is what configures the native stack trace capture (using internal engines like V8's Error.captureStackTrace) and populates the message property. Without calling super(message), these critical properties remain empty.

Q2: Why does try/catch fail to catch an error thrown inside a raw setTimeout callback, but it correctly catches a rejection from an awaited promise inside an async function? What's the structural difference in execution timing between the two?

Answer: A synchronous try/catch block wraps code execution in the current call stack frame. The setTimeout callback is pushed to Web APIs and executed in a completely new tick of the event loop. By the time that callback executes, the call stack containing the original try/catch has already finished executing and popped off the stack, leaving no handler active to capture the error.

In contrast, when you await a promise inside an async function, the engine compiles the code by suspending the function's execution context, storing it on the heap, and yielding control back to the event loop. When the promise resolves or rejects, the execution context is loaded back onto the call stack. If the promise is rejected, the engine resumes execution by re-throwing the rejection as a stack error inside the active evaluation frame, allowing the surrounding try/catch structure to catch it.

Q3: How do uncaught promise rejections behave in an Express route handler by default, and how should they be managed?

Answer: In Express (before version 5), async route handlers that encounter an uncaught promise rejection (e.g. throwing an error inside an async function) will fail silently or log as an unhandled rejection, but they will not automatically trigger the custom Express error handling middleware. The server will hang, leaving the HTTP request open until a gateway timeout occurs. To handle this properly, you must wrap all async route codes in a try/catch and explicitly call the next(err) callback, or use a routing wrapper utility (or libraries like express-async-errors) to automate this forwarding.

Q4: What is an unhandled promise rejection, and how can it be triaged globally in Node.js vs. the browser?

Answer: An unhandled rejection occurs when a Promise rejects but no rejection handler (like .catch() or a try/catch wrapping an await expression) is attached. In Node.js, you can intercept these globally using the process event process.on('unhandledRejection', (reason, promise) => { ... }). In browser environments, they are captured using the window event window.addEventListener('unhandledrejection', event => { ... }). Intercepting these globally is critical for logging telemetry data and clean process terminations under critical system failures.
