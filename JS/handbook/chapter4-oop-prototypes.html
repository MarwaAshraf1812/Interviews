Chapter 4: Object-Oriented Programming & Prototypes

Deconstruct JavaScript's unique prototypal inheritance mechanics: master the prototype chain lookup walk, analyze constructor function blueprints, demystify the ES6 class syntactic sugar, verify compilation behaviors, and explore runtime state protection techniques like seal and freeze.

01

The Prototype Chain & Lookup Walk

The Family Tree Analogy: Think of inheriting traits from your family tree. You might not personally "have" your grandmother's eye color coded into your own DNA copy — but if someone asks "what's your eye color?" and you don't have an answer yourself, JavaScript (like a genealogist) walks up your family tree until it finds an ancestor who does. That's the prototype chain — objects don't duplicate properties, they look them up through a chain of linked ancestors.

Why it exists: JavaScript needed a way for objects to share behavior (methods) without copying that behavior into every single instance. Copying methods to every instance wastes memory and makes updates painful (change one method, and you'd have to update every single object). The prototype chain lets many objects share one set of methods by linking to a common ancestor object.

Technically: Every object in JavaScript has an internal, hidden link to another object called its prototype (conceptually known as [[Prototype]]). When you access a property on an object, JS first checks the object itself. If not found, it walks up the prototype chain, checking each linked object in turn, until it finds the property or reaches null (the end of the chain). 

const animal = {
  eats: true,
  walk() { console.log("Animal walks"); }
};

const rabbit = {
  jumps: true,
  __proto__: animal // rabbit's prototype is animal
};

console.log(rabbit.eats);  // true — not on rabbit itself, found via prototype chain
rabbit.walk();             // "Animal walks" — method found on animal, not rabbit
console.log(rabbit.jumps); // true — found directly on rabbit
Under the Hood: The Lookup Walk
Every object eventually chains up to Object.prototype, which is why every object — no matter how it was created — has access to utility methods like .toString(), .hasOwnProperty(), and .valueOf(). 
console.log(rabbit.hasOwnProperty('jumps')); // true — own property
console.log(rabbit.hasOwnProperty('eats'));  // false — inherited, not own
Under the Hood: Constructor Functions & The "new" Keyword

When a function is called with the new operator, it serves as a constructor to create instances: 

function Animal(name) {
  this.name = name;
}
Animal.prototype.walk = function () {
  console.log(`${this.name} walks`);
};

const dog = new Animal("Rex");
dog.walk(); // "Rex walks"

What new actually does under the hood (Step-by-step):

  1. Creates a brand new empty object: {}
  2. Links this new object's internal [[Prototype]] to the constructor's prototype object (e.g. Animal.prototype).
  3. Invokes the constructor function with this bound to the newly created object.
  4. Returns the new object (unless the constructor function explicitly returns another non-primitive object).
💡 Mini Quiz: Prototype 'this' Context
If rabbit.walk is called, and walk is found on animal (rabbit's prototype), what does this refer to inside walk() — rabbit or animal? Why?
Answer: this refers to rabbit.

Explanation: In JavaScript, this is determined strictly at the call-site by how the function was invoked. Even though walk() is found on the prototype object (animal) and not on rabbit itself, the function is invoked as rabbit.walk(). According to the implicit binding rule, the object to the left of the dot (the receiver) becomes this. Walking up the prototype chain only resolves the method reference; it does not change the execution context's this binding.
02

__proto__ vs. .prototype

One of the absolute classic interview traps is confusing an object's internal prototype (usually referenced via the non-standard __proto__ accessor) with the .prototype property found on functions.

  • __proto__: This is an accessor property (a getter/setter on Object.prototype) that exposes the internal [[Prototype]] of an instance object. It represents the actual active link in the chain that JavaScript walks during property lookup. Every object in JavaScript has a __proto__ link.
  • .prototype: This is a normal property present only on functions (specifically constructors and classes). It does not represent the prototype of the function itself (which is Function.prototype). Instead, it is the object blueprint that will be assigned as the __proto__ of any instances created when calling that function with new.
function Dog() {}

// .prototype exists on the constructor function
console.log(Dog.prototype); // { constructor: Dog }

const rex = new Dog();

// Instances have __proto__, which points to the constructor's prototype blueprint
console.log(rex.__proto__ === Dog.prototype); // true

// Constructor function is itself an object, so it also has __proto__ (pointing to Function.prototype)
console.log(Dog.__proto__ === Function.prototype); // true
Trap: Confusing Reference Equality on Shared Methods
In prototype-based systems, methods declared on prototypes are shared by reference, not copied to each individual object. 
const a = {};
const b = {};
console.log(a.toString === b.toString); // true — exact same function reference
They point to the single implementation residing on Object.prototype.toString, saving major memory space.
03

Object.create vs. Object.assign

In modern JavaScript development, we frequently create new objects derived from existing ones. However, the mechanism under the hood changes drastically between Object.create and Object.assign.

  • Object.create(proto): Creates a brand new empty object, and sets its internal prototype link (__proto__) directly to the passed proto object. This is a live link. No properties are copied.
  • Object.assign(target, ...sources): Copies all enumerable own properties from one or more source objects into the target object. It returns the target object. This creates a one-time shallow copy snapshot. The two objects are completely disconnected afterward.
const animal = { eats: true };

// 1. Object.create (Prototypal Link)
const rabbit = Object.create(animal);
console.log(rabbit.hasOwnProperty('eats')); // false (inherited)
console.log(rabbit.eats);                  // true

// 2. Object.assign (Shallow Copy Snapshot)
const cat = Object.assign({}, animal);
console.log(cat.hasOwnProperty('eats'));    // true (copied as own property)
console.log(cat.eats);                      // true
Interview-Ready Formulation: The Mutation Difference
"Object.create(animal) sets animal as the new object's prototype — it's a live link, not a copy. So if you add or change a property on animal after creation, the new object sees that change immediately, because property lookups walk the prototype chain at access time. Object.assign({}, animal) copies animal's own properties as a one-time snapshot onto a new plain object — the two objects are completely disconnected afterward, so later changes to animal have zero effect on the copy."
💡 Mini Quiz: Live Lookup vs. Snapshot Copy
Given the following code, what does rabbit.legs and cat.legs output, and why? 
const animal = { eats: true };
const rabbit = Object.create(animal);
const cat = Object.assign({}, animal);

animal.legs = 4; // Add property to original animal
Output:
• rabbit.legs → 4
• cat.legs → undefined

Explanation: Since rabbit was created via Object.create(animal), it holds a live prototype link to animal. When we query rabbit.legs, the engine looks at rabbit, fails to find legs, walks the prototype chain to animal, and reads 4.
Since cat was created via Object.assign({}, animal), it only received a snapshot copy of the properties existing on animal at creation time (which was only { eats: true }). Later changes to animal have no impact on cat.
04

ES6 class Syntax (What It Really Is)

The Dashboard Analogy: Think of ES6 class syntax as a nicer dashboard on top of the same engine — the car (prototype-based inheritance) hasn't changed at all under the hood. You've just been given cleaner controls so you don't have to manually wire everything with Object.create and .prototype assignments yourself.

Why it was created: Before ES6, setting up inheritance required verbose, error-prone boilerplate. Developers coming from traditional class-based languages (Java, C++) found this model confusing. ES6 classes are syntactic sugar — they look like classical object-oriented programming, but compile down to the exact same prototype chain machinery. 

class Animal {
  constructor(name) {
    this.name = name;
  }
  walk() {
    console.log(`${this.name} walks`);
  }
}

console.log(typeof Animal); // "function" — classes ARE functions!
Under the Hood: Conceptual Desugaring

The engine compiles the above class syntax down to the following equivalent prototype constructor pattern: 

function Animal(name) {
  this.name = name;
}
Animal.prototype.walk = function () {
  console.log(`${this.name} walks`);
};
Key Engine-Level Differences (Not Just Cosmetic)
  • Classes are NOT hoisted like functions: While function declarations are fully hoisted and usable before their line of definition, classes behave like let/const and reside in the Temporal Dead Zone (TDZ) until the definition is reached. 
    const x = new Animal(); // ReferenceError: Cannot access 'Animal' before initialization
class Animal {}
  • Class bodies always execute in strict mode: Writing class definitions forces strict mode semantics internally. As a result, silent failures (like editing a frozen object) immediately crash with loud TypeError notifications.
  • Classes cannot be called without new: Standard constructor functions can accidentally be invoked as plain functions, setting properties on the global object or crashing. Classes prevent this by carrying an internal [[IsClassConstructor]] flag that triggers a hard exception. 
    function OldAnimal() {}
OldAnimal(); // No error (but produces a bug by setting 'this' incorrectly)

class NewAnimal {}
NewAnimal(); // TypeError: Class constructor NewAnimal cannot be invoked without 'new'
05

Inheritance (extends & super)

The Workshop Analogy: Think of extends like a child inheriting a family trade and a family workshop. They get all the tools their parent had, but they can also add their own specialized tools, or override how they use an inherited one. super is literally the phone line back to the parent — used to call the parent's version of something before (or instead of) doing your own thing.

Sharing behavior across related object types (e.g., a Dog and a Cat both being an Animal) without duplicating code is one of the most fundamental problems in OOP. extends sets up the prototype chain link automatically; super gives you controlled access to the parent's constructor and methods. 

class Animal {
  constructor(name) {
    this.name = name;
  }
  walk() {
    console.log(`${this.name} walks`);
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name); // MUST call super() before using 'this'!
    this.breed = breed;
  }
  bark() {
    console.log(`${this.name} barks`);
  }
  walk() {
    super.walk(); // Calls the parent Animal's walk method
    console.log(`${this.name} wags tail while walking`);
  }
}
Under the Hood: Subclass Constructor Order

In derived classes, why must super() be called before accessing this?

In a base class or old-style constructor function, this is created automatically by the new operator before any code runs. But in a derived class (one using extends), that automatic object-creation step does not happen in the subclass constructor.

Instead, this in the subclass constructor starts in an uninitialized state (similar to the Temporal Dead Zone). It does not exist yet. Only when super() runs does it invoke the parent constructor, which actually creates and initializes the object and returns it to the subclass. Trying to access this before super() throws a ReferenceError.

Trap: Missing Parent Arguments on super()
When overriding constructor methods in subclasses, ensure you forward all expected arguments to the parent constructor via super(...). If you forget to pass parameters that the parent relies on, parent-defined properties will end up undefined.
💡 Mini Quiz: super() Mechanics
What happens if you define a subclass constructor but omit the call to super() entirely?
Answer: The JS engine will throw a ReferenceError when trying to instantiate the class.

Explanation: Because the derived class constructor specifies that it extends a parent class, the engine assumes the parent is responsible for allocating the initial this binding. Without calling super(), this is never initialized. Even if you don't write any code referencing this in your constructor, JavaScript requires a derived class constructor to either call super() or return an object explicitly. If you omit it entirely, it crashes with a ReferenceError.

Object.freeze, Object.seal & Property Descriptors

The Filing Cabinet Analogy: Think of an object as a filing cabinet. Object.seal locks the cabinet so you can't add or remove drawers (properties) — but you can still edit what's written inside existing drawers. Object.freeze goes further: it locks the cabinet and glues the paper inside every drawer down — nothing can be added, removed, or edited.

JavaScript objects are mutable by default. Anyone holding a reference can add, delete, or change properties. To protect the integrity of configuration objects or API states, JavaScript provides control mechanisms that alter Property Descriptors. 

// Every property in JS is controlled by a hidden descriptor object
const obj = { x: 1 };
console.log(Object.getOwnPropertyDescriptor(obj, 'x'));
// { value: 1, writable: true, enumerable: true, configurable: true }
The Three Descriptor Flags
  • writable: Can the property's value be reassigned?
  • configurable: Can the property be deleted from the object, or can its descriptor configuration be changed later?
  • enumerable: Does the property show up when listing keys (e.g. Object.keys() or for...in)?
Compare: Seal vs. Freeze
Operation Add / Delete Props? Edit Values? Descriptor Impact
Object.seal() ❌ No ✔ Yes Sets configurable: false on all own properties.
Object.freeze() ❌ No ❌ No Sets both writable: false AND configurable: false. 
// Example of Object.freeze shallow limitation
const user = Object.freeze({
  name: "Marwa",
  address: { city: "Giza" }
});

user.name = "Changed";        // Fails silently (throws in strict mode)
console.log(user.name);        // "Marwa" (unchanged!)

user.address.city = "Cairo";   // WORKS! Nested objects are NOT protected
console.log(user.address.city); // "Cairo" (mutated!)
Trap: Freeze is Shallow & Silent Failures

Why did user.address.city change?
Object.freeze only sets property descriptors on the object's own direct keys. The key address is simply a pointer/reference. Freezing user prevents reassigning user.address = somethingElse, but the object that pointer reference references remains completely unfrozen.

The Solution (Deep Freeze Helper): To fully freeze nested objects recursively, you must write a custom deep freeze utility: 

function deepFreeze(obj) {
  Object.values(obj).forEach(val => {
    if (typeof val === 'object' && val !== null) {
      deepFreeze(val);
    }
  });
  return Object.freeze(obj);
}
07

Static Methods & Properties

The Factory Floor Analogy: Think of a class as a factory building. Instance methods are tools each worker (instance) carries around individually. Static methods are tools bolted to the factory floor itself — you don't need to hire a worker to use them; you just walk up to the building and use them directly.

Why they exist: Some functionality logically belongs to the class as a concept, not to any individual instance. This includes utility functions, factory methods, counters tracking all instances, and configuration constants. Static members let you attach that behavior directly to the class itself. 

class Animal {
  static count = 0; // static property — belongs to the class, not instances

  constructor(name) {
    this.name = name;
    Animal.count++;  // accessed via the class, not `this`
  }

  static createFeral() { // static method — a factory method
    return new Animal("Feral Animal");
  }
}

const a = new Animal("Rex");
const b = new Animal("Milo");
console.log(Animal.count); // 2 — tracked at the class level

const feral = Animal.createFeral();
console.log(feral.name);   // "Feral Animal"

// Critical: static members are NOT accessible on instances
console.log(a.count);       // undefined
console.log(a.createFeral); // undefined
Under the Hood: Attachment to Constructor Object

Static members are attached directly to the constructor function object itself, not to .prototype. That's the whole distinction: 

console.log(typeof Animal.createFeral);   // "function" — lives directly on Animal
console.log(Animal.prototype.createFeral); // undefined — NOT on prototype, so instances can't inherit it

Because instance property lookup only walks the prototype chain (instance.__proto__ → Constructor.prototype → Object.prototype), it never looks at properties sitting on the constructor function itself.

Trap: Referencing Statics Inside Instances Without Class Name
You cannot directly reference static variables inside instance methods without qualifying them with the class name or constructor reference: 
class Counter {
  static total = 0;
  increment() {
    total++; // ReferenceError: total is not defined!
    // Correct: Counter.total++ or this.constructor.total++
  }
}
💡 Mini Quiz: Static Method Resolution Failures
Why does a.createFeral() fail with TypeError: a.createFeral is not a function, even though Animal.createFeral() works fine? Walk through where static methods live.
Answer: Static methods are attached directly to the class constructor function object (Animal.createFeral), not to Animal.prototype.

Explanation: When you call a method on an instance (a.createFeral()), JavaScript's property lookup walks the prototype chain (a.__proto__ → Animal.prototype → Object.prototype). Since the static method resides directly on the Animal constructor function itself, and the constructor function is not part of the instance's prototype chain, it is completely invisible to the instance. The lookup resolves to undefined, and attempting to call undefined() throws a TypeError.

Getters & Setters

The Vending Machine Analogy: Think of a getter/setter like a vending machine's front panel — you press a button (access a property normally, obj.price) but behind the panel, real logic runs (calculations, validation, logging) before you get your result. From the outside, it looks like plain property access — no parentheses, no method call syntax — but real code executes underneath.

Why they exist: Sometimes you want a property to be computed dynamically on the fly, or you want to validate/transform a value whenever it's assigned, without forcing users of your class to use method-call syntaxes like obj.getPrice() or obj.setPrice(val). 

class Rectangle {
  constructor(width, height) {
    this.width = width;
    this.height = height;
  }

  get area() {                    // getter — accessed WITHOUT parentheses
    return this.width * this.height;
  }

  set area(value) {                // setter — assigned WITHOUT parentheses
    console.log("Adjusting width to match desired area");
    this.width = value / this.height;
  }
}

const rect = new Rectangle(4, 5);
console.log(rect.area); // 20 — invokes the getter function under the hood

rect.area = 50;         // invokes the setter function under the hood
console.log(rect.width); // 10 — recalculated by the setter
Under the Hood: Property Descriptors
Getters and setters are configured under the hood using special properties in the property descriptor: 
console.log(Object.getOwnPropertyDescriptor(Rectangle.prototype, 'area'));
// { get: [Function: get area], set: [Function: set area], enumerable: false, configurable: true }
Notice there is no value or writable key in the descriptor. The properties get and set take over these duties entirely.
Trap: Infinite Recursion & Stack Overflow

If your getter/setter tries to read or write the same property name it is defining, it triggers infinite recursion: 

class Bad {
  get name() {
    return this.name; // Calling this.name calls the getter again!
  }
}
const b = new Bad();
b.name; // RangeError: Maximum call stack size exceeded

The Fix: You must store the actual value in a backing property with a different name (such as prepending an underscore or using a private #name field): 

class Good {
  #name;
  get name() { return this.#name; }
  set name(v) { this.#name = v; }
}
💡 Mini Quiz: Getter Recursion
Why does writing get name() { return this.name; } cause an infinite loop / stack overflow?
Answer: The getter does not create a data storage box named "name"; it creates a function to execute whenever ".name" is accessed.

Explanation: When you call this.name inside the getter, JavaScript sees it as another access request for the property name. It resolves this request by invoking the getter function again. This creates a recursive loop of function calls (get name() calling get name()) without an exit condition, quickly exhausting the call stack and throwing a RangeError.
09

Private Class Fields (#field)

The Vault Analogy: Think of public properties like items lying on your desk, and private fields (#field) like contents locked inside a biometric vault. The outside world cannot bypass security — they can only see or modify the contents if you provide controlled pathways (getters/setters).

Why they exist: Before ES2022, JavaScript had no language-level support for true property privacy. Developers faked it using naming conventions like _email, which was only a social contract. Private class fields prefixed with # are enforced strictly by the V8/JavaScript engines, preventing external read/write completely. 

class User {
  #email;             // must be explicitly declared in the class body
  #loginAttempts = 0;

  constructor(email) {
    this.#email = email;
  }

  #logAttempt() {     // private methods are also supported
    this.#loginAttempts++;
  }

  login() {
    this.#logAttempt();
    console.log(`Attempt #${this.#loginAttempts}`);
  }
}

const u = new User("marwa@example.com");
u.login(); // "Attempt #1"

// Attempts to read private members directly throw or return undefined
console.log(u.email);       // undefined
console.log(Object.keys(u)); // [] — private properties are invisible to keys/looping
// console.log(u.#email);   // SyntaxError: Private field '#email' must be declared...
Under the Hood: Parse-time vs. Runtime Enforcement

Unlike conventions, private fields are compiled differently by the engine:

  • They are not stored in the regular property array of the object and are completely omitted from JSON.stringify or bracket lookup obj["#email"].
  • They are lexically scoped to the class body. The parser analyzes the source code at compile time, and writing u.#email outside the class scope immediately triggers a SyntaxError before the engine even executes the code.
Trap: Private Fields in Subclasses
Private properties are strictly encapsulated to the declaring class. Subclasses (via extends) cannot access them: 
class Admin extends User {
  checkEmail() {
    console.log(this.#email); // SyntaxError! Private fields are not inherited.
  }
}
💡 Mini Quiz: Convention vs. Syntactic Privacy
Why is referencing #email outside a class body a SyntaxError, while referencing _email works fine?
Answer: _email is a regular property governed by developer convention, while #email is a distinct syntax structure enforced by the parser.

Explanation: The underscore in _email carries no special meaning to the JavaScript engine; it is just a public property key that developer guidelines request you do not touch.
Conversely, the # character prefix represents private lexical identifiers. The JavaScript parser checks all references to # properties at parse time. Any reference to a private field outside of its declaring class body is grammatically invalid, triggering a SyntaxError and preventing execution entirely.

Polymorphism

The Universal Remote Analogy: Think of a universal remote control. Pressing the "power" button does something completely different depending on which device you are pointing it at (the TV turns on, the soundbar wakes up, or the projector rolls down). You interact with the exact same interface (the power button), but the receiver objects choose their own implementation of how to respond.

Why it exists: Without polymorphism, code that coordinates different classes would require complicated conditional switching blocks (e.g. if (isDog) speakWoof(); else if (isCat) speakMeow();). Polymorphism lets you program to a uniform interface, calling the same method name on different objects and trusting each to behave correctly. 

class Animal {
  speak() { console.log("Generic animal sound"); }
}

class Dog extends Animal {
  speak() { console.log("Woof!"); } // overrides Animal.speak
}

class Cat extends Animal {
  speak() { console.log("Meow!"); }
}

const animals = [new Animal(), new Dog(), new Cat()];
animals.forEach(animal => animal.speak());
// "Generic animal sound"
// "Woof!"
// "Meow!"
Under the Hood: Prototype Chains Enable Polymorphism
Polymorphism is not a separate engine mechanism; it is the natural consequence of the prototype chain lookup:
  • When the engine evaluates dog.speak(), it finds the overridden method directly on Dog.prototype first and executes it.
  • If a subclass does not override the method, the engine continues searching up the prototype chain until it hits the implementation on Animal.prototype.
Duck Typing in JavaScript

In dynamically-typed languages like JavaScript, polymorphism does not require objects to share a common class inheritance tree. This is called Duck Typing: "If it walks like a duck and quacks like a duck, it's a duck." 

const robot = { speak() { console.log("BEEP BOOP"); } };
const players = [new Dog(), robot];
players.forEach(p => p.speak()); // "Woof!", "BEEP BOOP" both work!

The loop executes .speak() successfully on both objects despite robot having no relation to Animal.

💡 Mini Quiz: Polymorphism & Base Classes
Does JavaScript require Dog and Cat to share a common base class (Animal) for polymorphism to work? What would happen if you removed extends Animal?
Answer: No, a shared base class is not required for polymorphism in JavaScript.

Explanation: Because JavaScript is a dynamically typed language, it uses duck typing. The invocation animal.speak() resolves successfully as long as the target object has a property named speak that is callable as a function. Removing extends Animal does not break the polymorphism in the loop; the engine only checks for the method's presence at runtime, regardless of the object's inheritance ancestors.

Composition vs. Inheritance

The Toolkit Analogy: Inheritance is like a family tree — you inherit a fixed bundle of traits from your parent, whether they are useful to you or not. Composition is like assembling a custom toolkit — you pick, choose, and pack only the specific tools you need for the job, without being constrained by any lineage.

Why this is a hot interview topic: Deep class inheritance (extends) builds rigid, tightly coupled taxonomies. If requirements change (or an exception arises), class structures break down, leading to the famous "fragile base class" problem. 

// The Fragile/Inflexible Inheritance Trap
class Bird {
  fly() { console.log("Flying!"); }
}
class Penguin extends Bird {
  // Penguins can't fly! But they are birds.
  // Overriding to throw or do nothing violates polymorphism interfaces
  fly() { throw new Error("Can't fly!"); }
}
The Solution: Composing Capabilities
Instead of classifying objects by what they are (inheritance), composition defines objects by what they do or have: 
const canFly = {
  fly() { console.log(`${this.name} is flying`); }
};
const canSwim = {
  swim() { console.log(`${this.name} is swimming`); }
};

function createDuck(name) {
  return Object.assign({ name }, canFly, canSwim); // Duck has both
}
function createPenguin(name) {
  return Object.assign({ name }, canSwim); // Penguin only gets swimming
}
Mixins: Class-based Composition

Since ES6 classes can only inherit from a single parent class, we can use Mixins (functions that return classes extending a passed base class) to compose behaviors: 

const CanFly = Base => class extends Base {
  fly() { console.log(`${this.name} flies`); }
};
const CanSwim = Base => class extends Base {
  swim() { console.log(`${this.name} swims`); }
};

class BaseAnimal {
  constructor(name) { this.name = name; }
}

class Duck extends CanSwim(CanFly(BaseAnimal)) {}
💡 Mini Quiz: Composition vs. Inheritance Trade-offs
In the Penguin/Bird example, what specifically goes wrong with using inheritance, and how does composition solve it structurally?
Answer: Inheritance forces an all-or-nothing capability bundle, whereas composition lets you assemble specific behaviors on demand.

Explanation: When using inheritance (class Penguin extends Bird), the subclass is forced to receive all methods defined on the parent class (like fly()). If a subclass cannot use a parent capability, you must override it to throw an error, which breaks uniform polymorphism interfaces and crashes callers.
Composition avoids this entirely because objects are built by copying behavior blocks. A penguin simply is not composed with canFly, meaning the method never exists on its prototype in the first place.

Interview Questions & Answers

Prepare for senior-level interviews with these conceptual questions and answers:

Q1: What does Object.create(parent) do, and how does it compare to Object.assign({}, parent)?

Answer: Object.create(parent) creates a new object and points its internal [[Prototype]] directly to parent. This sets up a live link; any later changes or additions to parent are immediately reflected on the child because lookup walks the prototype chain at access time. On the other hand, Object.assign({}, parent) copies the enumerable own properties of parent as a one-time shallow snapshot into a new object. The two objects are completely disconnected, so modifications to parent have no effect on the copy.

Q2: Why does invoking a class constructor without 'new' throw a TypeError, while regular constructors fail silently?

Answer: ES6 classes are not just cosmetic sugar; they introduce distinct runtime protections enforced by the engine. Class constructors are tagged with an internal [[IsClassConstructor]] metadata flag. At invocation time, the JavaScript engine checks this flag and throws a TypeError immediately if the new operator was omitted. Regular constructor functions do not have this flag, so when called without new, they execute like standard functions, binding this to the global object or undefined (in strict mode), which silently leads to bugs.

Q3: Why must super() be called first in a derived class constructor before using 'this'?

Answer: In base classes or old-style constructor functions, the engine automatically creates the object instance and assigns it to this before executing any constructor code. In derived (subclass) constructors, this automatic allocation is bypassed. The derived class constructor expects the parent constructor to instantiate the object. Therefore, this sits in an uninitialized state (similar to the Temporal Dead Zone) until super() executes, invokes the parent constructor, sets up the parent's properties, and initializes the instance. Attempting to read or write to this prior to calling super() results in a ReferenceError.

Q4: Why does Object.freeze fail to protect nested objects, and how do you resolve it?

Answer: Object.freeze operates shallowly: it only changes the property descriptors (setting writable: false and configurable: false) on the direct, top-level keys of the object. If a top-level key holds a reference to another object (like user.address), freezing only prevents the reference pointer itself from being reassigned. The object located at that reference address remains mutable. To protect nested data, you must implement a recursive deep freeze helper that walks the object tree, calling Object.freeze on every nested object.

Q5: Why does a.createFeral() fail with a TypeError, even though Animal.createFeral() works fine?

Answer: Static methods are attached directly to the constructor function object (Animal.createFeral), not to Animal.prototype. Instance property lookup only searches the prototype chain (a.__proto__ → Animal.prototype → Object.prototype). Since the constructor function itself is never part of that prototype chain, the static method is not reachable from the instance a. a.createFeral resolves to undefined, and attempting to invoke it throws a TypeError.

Q6: Why does writing get name() { return this.name; } trigger an infinite loop / stack overflow?

Answer: The getter does not create any actual storage for the property; it merely binds a function to be executed whenever the property name is accessed. Referencing this.name inside the getter body triggers another evaluation of the name property, recursively calling the same getter function indefinitely until the call stack limit is reached. A separate backing property (such as a private #name field) must be used to house the actual value.

Q7: What is the fundamental difference in how _email and #email are enforced as private fields?

Answer: _email is enforced solely by convention; it is a regular public property that other developers are politely asked not to mutate directly. #email is enforced by JavaScript's grammar rules. Syntaxes using the # prefix are invalid outside the class body, meaning any attempt to access #email externally is caught by the parser and throws a compile-time SyntaxError.

Q8: Does JavaScript require subclasses to share a common base class for polymorphism to work?

Answer: No. Since JavaScript is dynamically typed, it utilizes duck typing. The runtime only verifies whether a method exists and is callable when it is invoked. Two completely unrelated objects or classes can participate in polymorphism as long as they implement the expected method interface.

Q9: Why is composition preferred over inheritance in modern JavaScript architectures?

Answer: Inheritance establishes a rigid, all-or-nothing dependency model where subclasses must accept every parent capability, forcing awkward workarounds or exception handling when capabilities mismatch. Composition avoids this by assembling objects out of small, decoupled blocks of behavior on demand, resulting in much more flexible and refactor-friendly designs.
