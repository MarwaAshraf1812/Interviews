Chapter 3: Advanced React Patterns

Master modern composition paradigms: Compound Components for flexible UI libraries, legacy Render Props logic-sharing mechanics, and Higher-Order Components (HOCs).

01

Compound Components Pattern

Imagine a restaurant menu with fixed combos (e.g. "Burger + Fries + Drink", no substitutions) vs a build-your-own-bowl place where you pick each ingredient yourself, but the system (bowl → base → toppings → sauce) enforces a sensible structure.

Composition patterns are React's way of building the second kind — components that give consumers complete flexibility within a structural boundary, instead of one rigid, over-configured component with 15 boolean props (an anti-pattern known as prop explosion).

Instead of writing a monolithic component that controls everything internally, you expose several sub-components meant to be used together, sharing implicit state via React Context.

Real-World Reference: UI Libraries
This is exactly how state-of-the-art UI libraries like Radix UI, shadcn/ui, and Reach UI structure components — such as <Select>, <Select.Trigger>, <Select.Content>, etc. It allows developers to customize rendering layout and markup without breaking the component's internal behavior. 
import { createContext, useContext, useState } from 'react';

const ModalContext = createContext();

// 1. Parent wrapper defines provider and shared state
function Modal({ children, onClose }) {
  return (
    <ModalContext.Provider value={{ onClose }}>
      <div className="modal-overlay">{children}</div>
    </ModalContext.Provider>
  );
}

// Attach sub-components as static properties
Modal.Header = function ModalHeader({ children }) {
  const { onClose } = useContext(ModalContext);
  return (
    <div className="modal-header">
      {children}
      <button onClick={onClose}>×</button>
    </div>
  );
};

Modal.Body = function ModalBody({ children }) {
  return <div className="modal-body">{children}</div>;
};

Modal.Footer = function ModalFooter({ children }) {
  return <div className="modal-footer">{children}</div>;
};
Compound Component Implicit Context: Modal (Provider: onClose) ├── Modal.Header (Consumes: onClose) ├── Modal.Body (Children) └── Modal.Footer (Children)
02

Render Props Pattern

The Render Props pattern involves passing a function as a prop to a component, which the component calls to render its UI. The component owns the stateful logic (such as tracking coordinates, fetching data, or handling subscriptions) and delegates UI styling to the consumer.

Before React Hooks (pre-2019), render props was one of the standard patterns for sharing stateful logic. In modern React, custom hooks have largely replaced render props, but it is still vital to recognize it in older codebases. 

function DataFetcher({ url, render }) {
  const [data, setData] = useState(null);

  useEffect(() => {
    fetch(url).then(res => res.json()).then(setData);
  }, [url]);

  // The caller controls layout by passing a callback function
  return render(data);
}

// Usage:
<DataFetcher
  url="/api/orders"
  render={(data) => data ? <OrderList orders={data} /> : <p>Loading...</p>}
/>
03

Higher-Order Components (HOCs)

A Higher-Order Component (HOC) is a pure function that takes a component as an argument and returns a new, enhanced component. This is a functional programming pattern used for cross-cutting concerns (such as authorization, logging, or caching).

Similar to Render Props, HOCs were widely used pre-2019 to share stateful behaviors. In modern React, custom hooks are preferred because HOCs introduce wrapper nesting and can lead to prop namespace collisions. 

function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const { user } = useContext(UserContext);
    if (!user) return <p>Please log in</p>;
    
    // Pass down props and inject the user context
    return <WrappedComponent {...props} user={user} />;
  };
}

05

Interview Questions & Answers

Q1: What is 'prop explosion' and how do compound components resolve it?

Answer: Prop explosion is the anti-pattern where components accept a bloated list of props to manage layout and sub-component visibility, making them fragile and hard to maintain. Compound components resolve this by using a shared context to allow parent components to compose their children (like Modal.Header or Modal.Body) directly, placing control back into the caller's hands while keeping the internal state hidden.

Q2: Why are Render Props and Higher-Order Components largely obsolete in modern React?

Answer: Render Props and HOCs were introduced to share stateful logic across components. However, they both create "wrapper hell" (extra layers in the React component tree) and can lead to prop namespace collisions. Modern React uses Custom Hooks instead, which share stateful logic directly within the component body without nesting.

Q3: How do compound components share implicit state, and what is the risk of using child elements outside the parent container?

Answer: Compound components share implicit state via React Context (the parent acts as a Provider, and sub-components use useContext). The main risk of using a sub-component outside the parent container is that the context lookup will fail (resulting in undefined), which can cause runtime exceptions unless the sub-component explicitly guards against a null context.
