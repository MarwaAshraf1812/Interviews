01

Component vs Element vs Instance

In React, understanding the conceptual boundaries between a Component, an Element, and an Instance is a foundational requirement.

A Component is a function or class that acts as a blueprint or factory. A React Element is a lightweight, plain JavaScript object that describes what should appear on screen. It is immutable and contains the element type and properties. An Instance is the internal, runtime data structure (a Fiber node in modern React) instantiated to manage a component's lifecycle and local state. 

// 1. Component: The blueprint/factory
function CustomButton({ title }) {
  return <button>{title}</button>;
}

// 2. Element: The plain description object
const btnElement = <CustomButton title="Submit" />;
// Under the hood, btnElement is a simple JS object:
// { type: CustomButton, props: { title: "Submit" } }

// 3. Instance: The runtime Fiber node managing state
// React creates this internally during render to hold count state, refs, and link to the DOM node.
Under the Hood: React Element Structure
According to React internals, calling React.createElement or the modern JSX runtime returns a frozen object matching this structure:
  • $$typeof: A special Symbol (Symbol.for('react.element')) used to prevent XSS injection attacks.
  • type: A string (like 'div') or a reference to a constructor/function component.
  • key: A unique identifier for reconciling sibling nodes.
  • ref: A reference handle to the native DOM node.
  • props: An object containing attributes, classNames, and children.
Interview Trap

Trap: An interviewer asks: "Can you mutate a React Element's props directly to change what renders?"
Correct Answer: ❌ No. React Elements are strictly immutable. They are frozen using Object.freeze(element) or Object.freeze(element.props). If you want to change what's rendered, you must trigger a re-render to generate a new element tree.

Interview-Ready Answer
"A component is a template function or class. An element is a lightweight, immutable JavaScript description of a DOM element or component. An instance is the in-memory Fiber representation created by React's reconciler to track the component's state, props, and native DOM bindings."
💡 Mini Quiz
What is the value of $$typeof in a React element, and how does it prevent security exploits?
Answer: Symbol.for('react.element')

Explanation: If a database stores user-submitted JSON containing elements to render, a malicious user could submit a mock React element payload to cause an XSS injection. Since JSON cannot contain JavaScript Symbols, React checks element.$$typeof === Symbol.for('react.element'). If it is missing or invalid, React will refuse to render the object, preventing XSS exploits.
02

Virtual DOM vs React Fiber

Before React 16, the Stack Reconciler executed updates recursively and synchronously. Once it started, it could not yield the thread, blocking user interactions (like typing) during large renders.

React Fiber is a complete rewrite of the reconciliation architecture. It represents the component tree as a linked list of Fiber nodes. Because it acts as a virtual call stack, React can pause, resume, or discard rendering computations, prioritizing urgent browser events (like typing or animation frames) using cooperative scheduling.

Under the Hood: Fiber Nodes and Linked Lists
Unlike the old virtual tree which required recursive traversal, Fiber maps the tree to a singly linked list with three pointer directions:
  1. child: Points to the first immediate child.
  2. sibling: Points to the immediate next sibling.
  3. return: Points back to the parent (simulating a stack frame return pointer).
This enables React to pause parsing at any node, return control to the browser main thread, and resume precisely where it left off.
Linked List Navigation in Fiber: ┌──────────────┐ │ Parent Fiber│ └──────┬───────┘ │ child ▼ ┌──────────────┐ sibling ┌──────────────┐ │ Child Fiber1├──────────►│ Child Fiber2│ └──────┬───────┘ └──────┬───────┘ │ return │ return └───────────► Parent ◄─────┘

The Two Phases of Fiber Rendering

1. Render Phase (Asynchronous & Interruptible): React walks the fiber tree, evaluates component updates, computes diffs, and compiles a list of side-effects. This phase can be paused or restarted.
2. Commit Phase (Synchronous & Blocking): React applies the calculated changes to the DOM and runs layout/effect hooks. This phase must complete in a single synchronous pass to avoid visual layout glitches.

Interview Trap

Trap: "Since the Render phase is asynchronous and can run multiple times before committing, what is a common bug caused by putting side-effects (like fetch requests or mutations) inside the body of a functional component?"
Correct Answer: The render body runs during the interruptible Render phase. If rendering is paused or aborted, the body may run multiple times, causing duplicate fetch requests or data mutations. All side-effects must reside inside useEffect (which runs in the commit phase) or event handlers.

💡 Mini Quiz
Which phase of React rendering is interruptible, and why is the commit phase kept synchronous?
Answer: The Render Phase is interruptible.

Explanation: The commit phase is kept synchronous because it performs actual DOM updates. If it were asynchronous or interruptible, the browser might render a partially updated page structure, resulting in visual stuttering and inconsistent layouts.
03

Reconciliation & Diffing Heuristics

Reconciling two arbitrary tree structures has a computational complexity of O(n³). To make rendering fast, React employs a heuristic algorithm with O(n) complexity.

Under the Hood: The O(n) Heuristic Hypotheses
React's O(n) diffing relies on two rules:
  1. Different Types: Two elements of different component/DOM types will produce different trees. React destroys the old subtree entirely and creates the new subtree from scratch (unmounting old nodes, running cleanups, and mounting new elements).
  2. Sibling Keys: Sibling elements can be tagged with a stable key prop, hinting to the reconciler that they are stable across renders. Instead of remounting nodes, React reorders them.
// Scenario 1: Different Element Types (Subtree destroyed and rebuilt)
// Old: <div><Counter /></div>
// New: <section><Counter /></section> (Counter is unmounted and remounted!)

// Scenario 2: Same Element Type, Different Attributes (Attributes mutated)
// Old: <div className="active" id="1" />
// New: <div className="inactive" id="1" /> (DOM node preserved, attributes changed)
Interview Trap: Using Array Index as Key

Trap: "Why is it bad to use the array index as a key prop for lists?"
Correct Answer: Sibling elements are matched by their key. If you delete the first item in a list of three elements:
• Old: { key: 0, text: 'A' }, { key: 1, text: 'B' }, { key: 2, text: 'C' }
• New: { key: 0, text: 'B' }, { key: 1, text: 'C' }
React matches by key: key 0 (originally A) gets updated with B's props. Key 1 (originally B) gets updated with C's props. Key 2 is removed. This causes state matching bugs (such as input values staying on the wrong row) and breaks optimization.

💡 Mini Quiz
What happens to a stateful component when its type changes (e.g. from <div> to <span>)?
Answer: The component's state is completely destroyed.

Explanation: React tears down the old component tree completely, unmounting it from the DOM, which wipes out all local state and ref bindings.
04

JSX Compilation & Modern Transform

Browsers cannot execute JSX code directly. A compiler like Babel or SWC translates JSX markup into plain JavaScript functions.

Under the Hood: React.createElement vs Modern JSX runtime
1. Classic Compiler (React 16 and older):
Compiles JSX elements into React.createElement(type, props, ...children).
• Tradeoff: Requires import React from 'react' to be declared in every file.
• Tradeoff: Children arguments are passed as separate parameters, making performance optimization difficult.

2. Modern Compiler (React 17+):
Compiles JSX elements into direct calls to the JSX runtime (like _jsx) imported automatically.
• Advantage: No need to import React manually.
• Advantage: Children are passed as part of the props object, allowing for cleaner optimizations. 
// Input JSX:
const el = <div className="badge">React</div>;

// Classic Compilation (React 16):
const elClassic = React.createElement('div', { className: 'badge' }, 'React');

// Modern Compilation (React 17+):
import { jsx as _jsx } from 'react/jsx-runtime';
const elModern = _jsx('div', { className: 'badge', children: 'React' });
Interview-Ready Answer
"JSX is a syntactical sugar that is compiled into standard JavaScript. In earlier versions of React, JSX was compiled to React.createElement. React 17 introduced a modern JSX transform that imports compiler entry points directly from react/jsx-runtime, so we don't have to manually import React just to use JSX."
05

State vs Props & Batching Mechanics

Props are read-only configuration attributes passed down from parent components. State is mutable data managed locally inside a component.

When you invoke a state setter, React queues a re-render. To avoid multiple re-renders when setting multiple state variables, React implements Batching.

Under the Hood: React 18 Automatic Batching
Before React 18, state updates were only batched inside React event handlers. Updates triggered inside promises, setTimeout blocks, or native browser listeners did not batch, triggering multiple renders.
React 18 introduces Automatic Batching, aggregating state updates across all asynchronous boundaries. 
// Updates inside event handlers or timeouts:
const [count, setCount] = useState(0);
const [flag, setFlag] = useState(false);

const handleClick = () => {
  setCount(c => c + 1);
  setFlag(f => !f);
  // React 18: Batched together -> Only 1 render triggered!
};
Interview Trap

Trap: "What will the console output be immediately after setting state?" 

const [val, setVal] = useState(0);
const click = () => {
  setVal(1);
  console.log(val);
};
Correct Answer: 0. State updates are queued and batched, meaning they are asynchronous relative to the current calling scope. The local variable val does not update until the next render phase.

06

SyntheticEvent & Event Delegation

React wraps the browser's native events in a cross-browser SyntheticEvent object to guarantee consistent API properties across all platforms.

Instead of attaching event listeners to individual DOM nodes, React uses Event Delegation. In React 17+, event listeners are delegated to the React root DOM container (e.g. <div id="root">) instead of the global document object.

Under the Hood: React 16 vs React 17 Delegation
In React 16, events bubbled all the way to document before being processed. This caused issues when nesting multiple React instances or when integrating with non-React libraries that stopped event propagation.
React 17 attaches event listeners to the mount container root node, allowing sub-apps to stop events from escaping and isolating nested environments safely. 
function Form() {
  const handle = (e) => {
    e.preventDefault(); // Normalized SyntheticEvent method
    console.log(e.nativeEvent); // underlying browser Event object
  };
  // The click listener is NOT attached to the button itself,
  // but delegated to the mount root node.
  return <button onClick={handle}>Submit</button>;
}
Interview Trap

Trap: "What was Event Pooling in React 16, and why was it removed in React 17?"
Correct Answer: In React 16, SyntheticEvent objects were pooled and reused across events to reduce memory allocation. Once an event handler executed, all properties on the event object were nullified. This meant you couldn't access event properties inside an asynchronous callback (e.g., setTimeout(() => console.log(e.target.value))) unless you called e.persist(). Because modern JS engine garbage collection is fast, event pooling was removed in React 17.

07

Interview Questions & Answers

Q1: How does React's reconciliation achieve O(n) complexity instead of O(n³)?

Answer: React achieves O(n) complexity by applying two heuristic hypotheses: First, two elements of different types will generate different trees; React tears down the old tree and mounts the new one from scratch. Second, sibling elements can be tagged with stable key props so they can be matched and reordered instead of remounted.

Q2: What is React Fiber and how does it support Concurrent rendering?

Answer: React Fiber is a linked-list-based reconciliation architecture. Each fiber node represents a virtual call stack frame. This enables React to pause, resume, or discard rendering work, letting high-priority updates (like input events) execute first and yielding control back to the browser main thread to keep the page responsive.

Q3: What is the difference between classic and modern JSX compilation?

Answer: Classic compilation converts JSX to React.createElement calls, requiring React to be in scope. Modern compilation (React 17+) translates JSX directly into calls to internal runtime helpers (like _jsx from react/jsx-runtime) injected by the compiler, removing the manual React import requirement.

Q4: What is Event Delegation in React 17 and why is it preferred over React 16 delegation?

Answer: React delegates events to a single listener rather than attaching them to every DOM element. In React 16, events bubbled up to the global document node. In React 17+, event listeners are bound to the mount root DOM container, which prevents event propagation bugs when nesting multiple React applications.
