Chapter 2: Hooks & Closures

Detailed study of side effects, fiber hooks structures, closures under the hood, dependency arrays, cleanup workflows, and real-world socket/countdown timers.

01

useEffect: Why It Exists

Think of a React component as a recipe that only knows how to describe a dish (the JSX/render output). But sometimes you need to do things that aren't "part of the dish" — like calling the supplier to order ingredients (fetching data), setting a kitchen timer (subscriptions/intervals), or writing to a logbook (updating document.title, localStorage, etc.).

These are called side effects — anything that reaches outside the component to interact with the outside world (network, browser APIs, timers, subscriptions), rather than just computing what to render.

useEffect is React's designated place to say: "After you've rendered the UI, go do this side effect."

Under the Hood: Pure Functions vs Side Effects
Function components are supposed to be pure — same props/state in, same JSX out, no side effects during render. If you fetch data directly inside the function body, it will run on every single render. Since data updates trigger a state change and a subsequent re-render, this quickly results in an infinite loop. useEffect delays execution until after paint, resolving this issue. 
// ❌ Infinite Loop Trap: Runs on EVERY render, fetching and triggers a state update!
function OrdersList() {
  const [orders, setOrders] = useState([]);
  
  fetch('/api/orders')
    .then(res => res.json())
    .then(data => setOrders(data));
    
  return <ul>{/* ... */}</ul>;
}

The Three Dependency Array Modes

The dependency array directs React on when to re-trigger the effect:

  • No array: Runs after every single render.
  • Empty array ([]): Runs once, after the initial mount only (simulates componentDidMount).
  • Array with values ([dep1, dep2]): Runs after initial mount and subsequently only when any value in the dependency list changes.

Execution Flow: Render 1 (initial mount) ──► Paint JSX ──► Run useEffect Callback │ [Props or State Change] ◄─────────────────────────┘ │ Render 2 ──► Update JSX ──► Run Cleanup from Render 1 ──► Run useEffect Callback 2

Cleanup Functions — The Memory Leak Prevention Shield

Returning a function from your effect callback sets up a cleanup step. This cleanup runs right before the effect re-runs (if dependencies changed) and when the component unmounts.

In a live dashboard like FoodLoop, forgetting to clean up active listeners (like Socket.IO subscriptions or intervals) means these tasks keep executing in the background even after the user leaves the page—creating major memory leaks. 

// ✅ Correct pattern with subscription cleanup
useEffect(() => {
  const socket = connectToOrderUpdates(orderId);
  socket.on('statusChange', handleStatusChange);

  return () => {
    socket.off('statusChange', handleStatusChange);
    socket.disconnect(); // Prevents duplicate active connections
  };
}, [orderId]);
Case Study: Expiry Countdown Component

Scenario: You are building a component to display the time remaining before a food item expires. It needs to fetch the expiry date once from the API and start a 1-second interval to update the countdown text. 

import { useState, useEffect } from 'react';

function ExpiryCountdown({ productId }) {
  const [timeRemaining, setTimeRemaining] = useState(null);

  useEffect(() => {
    let isMounted = true;
    let intervalId;

    fetch(`/api/products/${productId}`)
      .then(res => res.json())
      .then(data => {
        if (!isMounted) return;
        const expiryDate = new Date(data.expiryDate);

        const updateCountdown = () => {
          const now = new Date();
          const difference = expiryDate - now;

          if (difference <= 0) {
            setTimeRemaining("Expired");
            clearInterval(intervalId);
            return;
          }

          const seconds = Math.floor((difference / 1000) % 60);
          const minutes = Math.floor((difference / 1000 / 60) % 60);
          const hours = Math.floor((difference / (1000 * 60 * 60)) % 24);
          const days = Math.floor(difference / (1000 * 60 * 60 * 24));

          setTimeRemaining(`${days}d ${hours}h ${minutes}m ${seconds}s`);
        };

        updateCountdown();
        intervalId = setInterval(updateCountdown, 1000);
      });

    return () => {
      isMounted = false;
      if (intervalId) clearInterval(intervalId);
    };
  }, [productId]);

  return <div>Time remaining: {timeRemaining ?? 'Loading...'}</div>;
}

Why [productId] is the dependency array: If the array were empty ([]), the component would only set up the countdown for whichever product was mounted first. If the parent component changed the productId prop later, the hook would never re-trigger, resulting in stale data. Including [productId] ensures that as soon as the ID changes, the old timer is cleared and the component fetches the new product's details.

💡 Mini Quiz
What two events trigger a useEffect's cleanup function to execute?
Answer: 1. Right before the effect callback re-runs (if dependencies changed).
2. When the component unmounts from the screen.
02

useRef: DOM Access & Persistent Values

While useState manages state variables that determine what gets painted on the screen (triggering a re-render when they change), useRef lets you remember values across renders without triggering a re-render.

Think of useState as a public whiteboard—whenever you write on it, everyone looks up and React re-renders. Think of useRef as a private sticky note in your pocket—you can read and write to it freely at any time, but it won't trigger a visual update.

The Two Core Use Cases

  1. Direct DOM Access: Accessing real HTML nodes directly (e.g., focusing an input, playing media, or triggering actions on hidden elements).
  2. Mutable Instance Store: Holding mutable values that must persist across render cycles but shouldn't trigger re-renders when updated (e.g., storing a setInterval timer ID so it can be cleared later).
Under the Hood: The Immutable Ref Object
Calling useRef(initialValue) returns a plain JavaScript object: { current: initialValue }. React guarantees that this object is exactly the same reference on every single render. You can mutate its .current property directly, but since React only tracks state updates via useState/useReducer, mutating a ref never schedules a re-render. 
// Use Case: Styled button triggering hidden file input
import { useRef } from 'react';

function BulkUpload() {
  const fileInputRef = useRef(null);

  const handleBtnClick = () => {
    // Escape the declarative model and click the hidden DOM node directly!
    fileInputRef.current.click();
  };

  return (
    <div>
      <input
        type="file"
        ref={fileInputRef}
        style={{ display: 'none' }}
        onChange={(e) => processFiles(e.target.files)}
      />
      <button onClick={handleBtnClick} class="styled-btn">
        Upload CSV File
      </button>
    </div>
  );
}
Gotcha: Reading/Writing refs during Render is a Violation
React assumes the render phase is a pure calculation. Mutating or reading ref.current during render (e.g., ref.current++ in the component body) makes the render impure. In Concurrent Mode or Strict Mode, React may render components multiple times or abort rendering midway, leading to unpredictable ref values and visual bugs. Always keep ref updates inside event handlers or effects.
💡 Mini Quiz
In FoodLoop's Bulk Upload feature, why would using useState instead of useRef to store the file input element be a bad choice?
Answer: 1. Unnecessary Re-renders: When the element is mounted, setting state would trigger an extra, useless render cycle because we only need the ref to trigger the file dialog imperatively, not to update any UI elements in the JSX.
2. Declarative vs Imperative: Opening a file dialog requires calling the imperative .click() DOM method. A state hook doesn't provide a direct handle to the DOM node, making direct DOM interaction impossible without query selectors, which breaks React's isolated component scope.
03

useMemo & useCallback: Memoization Hook Pair

Memoization is an optimization technique used to speed up computer programs by storing the results of expensive function calls and returning the cached result when the same inputs occur again. React provides two hooks to do this:

  1. useMemo: Caches the computed value returned by a function.
  2. useCallback: Caches the function definition reference itself.
The Analogy: Sticky Notes vs Brand New Objects
Imagine you have an expensive calculator (heavy computation) in your kitchen. Every time anyone opens the kitchen door (component re-renders), you re-run the entire calculation from scratch—even if the inputs to the calculation haven't changed.

useMemo is like writing the calculation result on a sticky note and sticking it to the calculator. You only recalculate if the actual inputs change.

useCallback does the same thing, but for function references. In JavaScript, functions are objects, so a newly declared function in a component body gets a fresh memory address on every single render. useCallback ensures the function retains the exact same reference between render passes. 
import { useMemo, useCallback } from 'react';

// 1. useMemo Caching computed total EGP value
function OrderSummary({ items }) {
  const total = useMemo(() => {
    return items.reduce((sum, item) => sum + item.price, 0);
  }, [items]);

  return <p>Total: {total} EGP</p>;
}

// 2. useCallback Caching function reference to keep it identical
function ProductList({ products }) {
  const handleAddToCart = useCallback((productId) => {
    console.log('Adding', productId);
  }, []); // Empty deps = same function reference returned on every render

  return products.map(p => (
    <ProductCard key={p.id} product={p} onAdd={handleAddToCart} />
  ));
}
Gotcha: Caching has overhead. Do NOT overuse!
useMemo and useCallback are not free. They cost memory (storing the cache) and CPU cycles (comparing dependency arrays on every render). Memoizing a cheap operation like const doubled = useMemo(() => count * 2, [count]) actually makes performance worse due to the check overhead. Only use them for genuinely expensive calculations, or to maintain referential stability for props passed to React.memo child components.
💡 Mini Quiz
In FoodLoop's Dynamic Pricing Engine, you have an expensive chart component wrapped in React.memo: <PricingChart data={priceHistory} onPointClick={handlePointClick} />. If the parent component does not wrap handlePointClick in useCallback, what will happen?
Answer: 1. Referential Discrepancy: On every parent render, a new handlePointClick function reference is created.
2. React.memo Bypass: Since React.memo does a shallow equality comparison (===) on props, it sees that the onPointClick reference changed (e.g., Address A !== Address B).
3. Unnecessary Render: The chart will re-render on every parent render cycle, completely wasting the heavy computation cache of React.memo.
04

useContext & Prop Drilling

In React, data is typically passed top-down via props (prop drilling). If a piece of data is needed by many deeply nested components, passing it down through intermediates that do not need it becomes highly verbose and error-prone.

The Context API solves this by creating a global bulletin board. A Provider component stores the value, and any child component at any depth can consume it directly using the useContext hook, skipping intermediate components.

Under the Hood: Resolution & Subscription
When you call useContext(MyContext), React walks up the fiber node tree at read-time to find the nearest matching MyContext.Provider above the component, and subscribes to its value.

⚠️ The Performance Trap: Whenever the provider's value changes, React automatically re-renders all components that consume that context. If the provider's value is an inline object literal (e.g. value={{ user, theme }}), it gets a new reference on every render, causing all consumers to re-render even if the properties did not change. Always wrap context values in useMemo! 
import { createContext, useContext, useState, useMemo } from 'react';

const UserContext = createContext(null);

function App() {
  const [user, setUser] = useState({ name: 'Marwa' });
  
  // Caching the context value ensures consumers only re-render when user actually changes
  const value = useMemo(() => ({ user, setUser }), [user]);

  return (
    <UserContext.Provider value={value}>
      <Dashboard />
    </UserContext.Provider>
  );
}

function UserBadge() {
  // Accesses the context directly, skipping intermediate components
  const { user } = useContext(UserContext);
  <p>Logged in as: {user.name}</p>;
}
05

Interview Questions & Answers

Q1: Why can't you perform side effects directly in the body of a functional component?

Answer: The body of a functional component runs during the React Render phase, which must remain a pure function of props and state. Placing side-effects here causes them to run on every render. If the side-effect triggers a state update, it will cause another render, throwing the component into an infinite render loop.

Q2: What is the difference between passing an empty array vs no array as dependencies?

Answer: Passing no array causes the effect to execute after every single render and paint cycle. Passing an empty array ([]) limits execution to once: immediately after the initial mount, and never again.

Q3: What are memory leaks in useEffect and how do you resolve them?

Answer: Memory leaks occur when effects establish persistent tasks (e.g. setIntervals, EventListeners, WebSockets) but fail to release them when the component unmounts. Resolving them requires returning a cleanup function from the effect callback that clears timers (clearInterval), removes event listeners (removeEventListener), or closes sockets.

Q4: When should you use useRef instead of useState?

Answer: Use useState when the value is directly rendered in the JSX or affects what appears on the screen (since changing state schedules a re-render to update the UI). Use useRef when you need to persist a mutable value across renders without triggering a re-render, or when you need direct access to a raw DOM node.

Q5: Why is reading or writing ref.current during rendering considered an anti-pattern?

Answer: Reading or mutating ref.current during the render phase is unsafe because React rendering must be a pure, side-effect-free calculation. React may render components concurrently or repeat renders before updating the DOM. Mutating a ref during render violates purity and can cause stale reads, race conditions, or erratic behaviors, particularly when React's concurrent mode features are active.

Q6: How do useMemo and useCallback differ conceptually?

Answer: useMemo is designed to cache the result of a calculation (a computed value) between render cycles. It runs the provided function and stores its output. useCallback caches the actual function reference itself. It does not call the function; it simply retains the memory address of the function declaration across renders to maintain reference stability.

Q7: Why does passing a non-memoized callback to a React.memo component break its optimization?

Answer: React.memo prevents a component from re-rendering only if its props are unchanged, verifying equality using reference comparison (===). Since standard JS functions are objects, defining an inline function or normal callback in the parent component creates a brand new function reference on every single parent render. Consequently, React.memo detects a "new" prop reference, bypasses the cache, and re-renders the child component anyway. Wrapping the callback in useCallback preserves the exact same reference and prevents this issue.

Q8: How does useContext resolve the value under the hood, and what triggers consumer components to re-render?

Answer: Under the hood, useContext walks up the Fiber tree starting from the calling component until it matches the nearest Provider node. When the value of the Provider changes (verified using Object.is reference comparison), React automatically marks all subscribing consumer components as dirty, triggering them to re-render.

Q9: Why is Context considered a poor fit for high-frequency state updates, and what should be used instead?

Answer: Context is not optimized for high-frequency updates because any change to the Provider's value forces every consuming component to re-render, which can bottleneck performance. For high-frequency state updates, developer-controlled granular subscription models (like Zustand, Redux, or Signals) are preferred because they allow components to select and subscribe only to specific slices of state.
